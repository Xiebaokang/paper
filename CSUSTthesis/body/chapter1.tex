\section{研究背景及意义}
% 近来以 Transformer 架构的大语言模型（Large Language Models, LLMs）在自然语言处理、计算机视觉及多模态学习等领域取得了突破性进展。从 BERT、GPT 系列到如今的 LLaMA、混合专家模型（Mixture of Experts, MoE）参数量与上下文长度不断攀升，使得深度学习任务的计算需求呈现爆炸式增长。在模型的推理部署阶段，性能表现直接关系到实时交互体验与服务成本。研究表明，Transformer 模型的核心计算负载主要集中在矩阵乘（Matrix Multiplication, Matmul）以及多头注意力机制（Multi-Head Attention, MHA）中。尽管矩阵乘在算力密集的硬件上已有较成熟的优化，但 MHA 由于涉及多个算子的链式组合且具有高度复杂的访存模式，已成为制约大模型推理延迟与吞吐量的核心瓶颈。
Transformer 架构的大语言模型（Large Language Models, LLMs）近来在自然语言处理、计算机视觉、多模态学习等方向取得了重要进展。从 BERT、GPT 系列模型演变为现在的 LLaMA 和混合专家模型（Mixture of Experts, MoE），参数量和上下文长度在持续增加，这让深度学习任务的计算需求表现出爆炸式的增长。模型在推理部署过程中体现出的性能水平，会影响到实时交互体验及服务成本。研究证明，Transformer 模型的主要计算负载集中于矩阵乘、多头注意力机制中。虽然矩阵乘在算力密集的硬件上已经有比较成熟的优化，但多头注意力由于涉及多个算子的链式组合并且具有高度复杂的访存模式，已经变成限制大模型推理延迟和吞吐量的主要瓶颈。

% 为了加速注意力机制，业界涌现了许多优秀的底层优化方案，通过算子融合（Operator Fusion）与分块计算（Tiling）显著提升了访存效率。然而，当前的加速成果高度依赖于 NVIDIA 的 CUDA 生态及其专有的硬件特性（如 Tensor Core、异步拷贝指令等）。与此同时，国内外异构计算加速设备层出不穷，尤其是以海光 DCU、寒武纪 NPU 等为代表的国产芯片正在快速崛起。由于不同硬件架构的指令集、存储层次及线程调度模型存在显著差异，现有厂商提供的底层高性能库（如 cuBLAS、rocBLAS）通常呈现“烟囱式”开发模式。开发者往往需要针对特定平台投入巨大的精力进行手工代码编写与性能调优，这种高度绑定的开发模式导致了严重的软件生态碎片化，极大地提高了算法跨平台迁移的成本与维护难度。
为加快注意力机制，行业出现了许多好的底层优化方法，通过算子融合（Operator Fusion）和分块计算（Tiling），明显提高访存效率。不过，当前加速的实现主要依靠 NVIDIA 的 CUDA 生态系统和其专有硬件特点，包括 Tensor Core、异步拷贝指令等。同时，国内外持续推出多种异构计算加速设备，海光 DCU 和寒武纪 NPU 等国产芯片的发展也很快。在不同的硬件架构在指令集、存储层次以及线程调度模型等方面有着较大区别，像 cuBLAS 和 rocBLAS 等现有厂商提供的底层高性能库，通常采用这种烟囱式的开发模式。因为这样开发者往往需要为特定平台花费大量时间进行代码的编写和性能调优的工作。因此这也带来了严重的软件生态碎片化问题，让算法跨平台迁移的成本和维护难度都变大了。

虽然解决这些问题的主要方式可以是自动化编译技术，但是在适配 Transformer 类算子和异构硬件时，现有的深度学习编译器仍然存在“语义鸿沟”和“控制力不足”的困难。例如 TVM 这类传统编译器大多使用基于循环的低级中间表示，它们所表示的张量计算的块级语义在 Lowering 的过程中丢失，这就使得编译器不容易分析得到线程块级别的优化机会。后来 Triton 这一类新型编译器虽然引入了块级抽象，但是它们主要依赖隐式的布局推断和黑盒化的数据搬运方式。面对海光 DCU 等拥不同硬件特征和存储结构的国产硬件时，该隐式机制通常不能生成最优的访存指令，并且开发者难以通过显式方式干预数据通路来消除硬件瓶颈。所以，研究一种可以保留高层块级语义，同时显式解耦计算与数据搬运、以达到精细化硬件控制的算子生成新方法，成为现在需要解决的重要问题。

针对硬件异构化、软件生态碎片化、现有编译技术局限性的多重挑战，使得研究高效的跨平台算子自动代码生成技术具有较大的科学意义和应用价值。传统编译器在块级逻辑表达和底层访存控制上有所欠缺，本文通过多级中间表示 MLIR 的多级抽象和方言机制，形成了一套可以把计算逻辑和数据搬运进行解耦的编译模式。通过引入显式的块级中间表示，在保留高层算子语义完整性的同时实现了对底层硬件特性的精确抽象和直接映射，并且使跨平台可移植性方面的异构硬件适配和扩展的工程复杂度得到明显降低。

% 传统的 MHA 实现方式通常遵循“计算与访存分离”的模式，即首先计算查询向量与键向量的点积以得到注意力分数矩阵，将其写入高带宽全局显存（HBM），随后读取该矩阵进行 Softmax 归一化操作，最后再次写入显存。这种频繁的显存读写操作引发了严重的内存墙\cite{gholami2024ai}（Memory Wall）问题，导致 GPU 的计算单元因长期等待数据而处于闲置状态，实际吞吐量远低于硬件理论峰值。
% 为了解决这一问题，FlashAttention 系列\cite{dao2022flashattention, dao2023flashattention, shah2024flashattention, zadouri2026flashattention}算法应运而生，通过分块计算与重计算策略彻底重塑了 MHA 的执行范式。第一代 FlashAttention\cite{dao2022flashattention} 利用 GPU 片上高速共享内存（SRAM）作为缓存，通过算子融合技术将矩阵乘、Mask 操作、Softmax 以及 Dropout 等多个步骤在片上一次性完成，避免了中间大矩阵对全局显存的反复读写，成功将访存复杂度从二次方降低为线性。随后的 FlashAttention-v2\cite{dao2023flashattention} 进一步优化了线程块层面的任务划分与并行策略，大幅减少了非矩阵乘操作带来的同步开销，并通过优化线程束级的数据布局有效缓解了共享内存的 Bank 冲突，显著提升了在 NVIDIA Ampere 架构\cite{nvidia_ampere}上的资源占用率。FlashAttention-v3\cite{shah2024flashattention} 则针对 NVIDIA Hopper 架构\cite{nvidia_hopper}特性进行了深度定制，通过利用 WGMMA 矩阵指令集与张量存储加速器（Tensor Memory Accelerator, TMA）等硬件特性\cite{luo2024benchmarking}，实现了计算流水线与访存流水线的深度重叠\cite{soi2025optimal}，进一步逼近硬件性能极限。最新的 FlashAttention-4\cite{zadouri2026flashattention} 是针对 NVIDIA Blackwell GPU 架构\cite{nvidia_blackwell}进行了算法与内核流水线协同设计的注意力优化算法，通过利用更大tile、软件模拟指数函数、条件性 Softmax 重缩放及 2-CTA 模式等技术，有效解决了该架构上非矩阵乘单元成为性能瓶颈的问题，从而实现了显著的性能提升。

传统的 MHA 实现方式为“计算与访存分离”的模式。它首先计算查询向量（Query）与键向量（Key）的点积得到注意力分数矩阵，在将其写入高带宽全局显存中（HBM），然后再读取该矩阵进行 Softmax 归一化操作，最后再次写入显存。这种频繁的显存读写操作造成了严重的内存墙\cite{gholami2024ai}（Memory Wall）问题。GPU 的计算单元因长期等待数据而处于闲置状态，导致实际吞吐量远低于硬件理论峰值。

为解决这一问题，FlashAttention 系列算法\cite{dao2022flashattention, dao2023flashattention, shah2024flashattention, zadouri2026flashattention}被提出。它借助分块计算和重计算策略彻底改变了 MHA 的执行方式。首先是FlashAttention-v1\cite{dao2022flashattention}，它把 GPU 片上高速共享内存（Shared Memory）作为缓存，运用算子融合技术在片上一次性完成矩阵乘、Mask 操作、Softmax 、 Dropout 等多个步骤。这样做防止中间大矩阵对全局显存反复读写，将访存复杂度从二次方降为线性。然后就是 FlashAttention-v2\cite{dao2023flashattention}， 它进一步优化了线程块方面的任务划分与并行策略，大量减少了非矩阵乘操作带来的同步开销。它通过优化线程束级的数据布局有效减轻了共享内存的 Bank 冲突。正因如此使得 MHA 的计算在 NVIDIA Ampere 架构上的资源占用率有了明显的提高。其次还有 FlashAttention-v3\cite{shah2024flashattention}，它针对 NVIDIA Hopper 架构\cite{nvidia_hopper}特性做了专门定制，它利用 WGMMA 矩阵指令集与张量存储加速器（Tensor Memory Accelerator, TMA）硬件特性\cite{luo2024benchmarking}，促使计算流水线与访存流水线的深度重叠\cite{soi2025optimal}，更接近硬件性能极限。最后就是 FlashAttention-v4\cite{zadouri2026flashattention}，它又针对 NVIDIA Blackwell 架构\cite{nvidia_blackwell}设计了算法与内核流水线协同的注意力优化算法，它为了解决该架构上非矩阵乘单元成为性能瓶颈的问题，采用了更大 tile、软件模拟指数函数、条件性 Softmax 重缩放及 2-CTA 模式等技术，实现了性能进一步提升。

% 然而，这些顶尖的优化成果大多深度绑定 NVIDIA CUDA 指令集及其底层的计算资源分配模式。一旦部署环境切换至海光 DCU 或寒武纪 NPU 等国产异构平台，由于缺乏等效的高性能内核支持，模型性能往往会出现断崖式下跌。这种对特定硬件生态的深度依赖，使得 Transformer 模型的跨平台部署面临着巨大的代码重构成本与维护压力。
从 FlashAttention 系列的提出，我们可以发现，大部分极致优化都很大程度上与 NVIDIA CUDA 指令集、其底层的计算资源分配模式紧密相连。因为没有等效的高性能内核提供支持，如果将部署环境转换为海光 DCU 或者寒武纪 NPU 等国产异构平台，模型性能通常会出现急剧下降的情况。因此这种对于特定硬件生态的强烈依赖，致使 Transformer 模型在进行跨平台部署时，面临着高昂的代码重构成本、较大的维护压力。

% 综上所述，虽然以 FlashAttention 为代表的手工深度优化算子极大地推高了 Transformer 模型的推理性能天花板，但其高昂的迁移成本与封闭的生态特性，已难以适应当前异构计算硬件百花齐放的行业现状。在硬件架构快速迭代与模型算子日益复杂的双重压力下，单纯依赖人工针对特定硬件编写高性能内核的传统模式已难以为继\cite{cheng2025pipethreader, liu2026dr, liao2025kernelevolve}。如何摆脱对单一硬件厂商底层库的过度依赖，在保证算子极致性能的同时实现跨平台的自动化部署，成为了学术界与工业界共同探索的焦点。这也促使研究者将目光转向深度学习编译器技术\cite{JSJA202508003}，试图通过自动化的代码生成与优化路径来打破这一僵局。
虽然以 FlashAttention 作为代表性的手工深度优化算子，显著地提高 Transformer 模型的推理性能上限，然而其高昂的迁移成本、封闭的生态特性，已然无法契合当下异构计算硬件蓬勃发展的行业状况。当面临硬件架构快速更新换代、模型算子越来越复杂这两方面的压力时，仅仅依靠人工针对特定硬件去编写高性能内核的传统方式，已然难以为继\cite{cheng2025pipethreader, liu2026dr, liao2025kernelevolve}。究竟怎样能够摆脱对单一硬件厂商底层库的过度依赖，并且在确保算子具备极致性能的前提下，达成跨平台的自动化部署，这已然成为学术界与工业界共同探寻的关键要点。这也使得研究者们将视线投向了机器学习编译器技术\cite{JSJA202508003}，尝试借助自动化的代码生成、优化途径来打破这一困境。

% 尽管这种基于循环的表达方式具有极高的通用性，但它导致了严重的“语义丢失”问题。当编译器将矩阵乘法拆解为 $for (i, j, k)$ 的标量循环后，原本的矩阵计算语义被破坏。在面向现代 GPU 架构时，硬件厂商为了提升算力，引入了专用的矩阵加速单元。这些单元通常要求数据以特定的分块形状和布局加载。为了利用这些硬件特性，基于循环的编译器必须执行复杂的张量化（Tensorization）过程，即通过模式匹配（Pattern Matching）尝试从破碎的标量循环中重建出矩阵语义，再映射到硬件内建指令。这一过程极其脆弱，往往因为循环边界的微小变化或复杂的算子融合逻辑而匹配失败，导致生成的代码回退到低效的标量计算模式，无法释放硬件的峰值算力。
这种基于循环的表达方式通用性较高，却引发了比较严重的“语义丢失”情况。当编译器把矩阵乘法分解成 $for (i, j, k)$ 的标量循环后，原本矩阵计算的语义遭到了破坏。面对现代 GPU 架构时，硬件厂商为提高算力，引入了专用的矩阵加速单元。这些单元一般需要数据按照特定的分块形状和布局来加载。为了运用这些硬件特性，基于循环的编译器不得不执行复杂的张量化（Tensorization）过程，也就是通过模式匹配（Pattern Matching）试着从破碎的标量循环中重塑矩阵语义，然后映射到硬件内建指令。这一过程很是脆弱，常常由于循环边界的细微变化或者复杂的算子融合逻辑而匹配失败，致使生成的代码退回到低效的标量计算模式，无法发挥硬件的峰值算力。

% 为了解决标量循环无法自然映射到矩阵加速单元的问题，以 OpenAI 推出的 Triton 为代表的新一代领域特定语言（Domain Specific Language, DSL）应运而生，并迅速成为 PyTorch\cite{paszke2019pytorch} 2.0 的默认后端技术。如图~\ref{1_3}~(b)，Triton 摒弃了细粒度的标量循环描述，转而采用块级的编程模型。在 Triton 中，基本的操作单元不再是单个浮点数，而是参数化的块。开发者使用 tl.load、tl.dot、tl.store 等原语直接对张量块进行操作。这种设计使得编译器能够天然地保留矩阵运算的语义，从而更容易地映射到 Tensor Core\cite{bhaskaracharya2020automatic} 等块级硬件指令上。此外，Triton 采用类似 Python 的语法，大幅降低了高性能算子的开发门槛，使得算法工程师能够以较高的抽象层次编写 GPU 内核。

% 尽管 Triton 在 NVIDIA GPU 上表现优异，但其设计哲学中包含的大量隐式推断机制，在面对跨平台异构部署时成为了新的阻碍。在数据布局与访存方面，Triton 倾向于自动管理共享内存的数据布局（Layout）和存取模式（Swizzle）\cite{zhou2026linear}。虽然它提供了 convert\_layout 等接口，但具体的内存排布逻辑对开发者是半透明的“黑盒”。在 NVIDIA 架构上，这套启发式策略运行良好，但是当迁移至海光 DCU 或其他国产 AI 芯片时，由于这些硬件具有不同的 Cache Line 大小、Bank 数量及冲突模式，Triton 的隐式策略往往无法生成最优的无冲突访问模式。开发者难以像在 CUDA 或汇编中那样，通过位运算显式地控制每一个数据的物理存储位置，导致性能在异构平台上出现严重衰退。而在调度策略方面，Triton 的许多编译器优化 Pass 是针对 NVIDIA 硬件特性深度定制的。在移植到其他架构时，往往需要重写大量后端逻辑。这种架构特定优化与语言通用性之间的矛盾，限制了其在广泛异构算力环境中的普及。
虽然 Triton 在 NVIDIA GPU 上有着出色表现，然而其设计理念里所包括的诸多隐式推断机制，在面临跨平台异构部署状况时，变成了新的阻碍因素。在数据布局、访存\cite{zhou2026linear}这两方面，Triton 偏向于对共享内存的数据布局和存取模式进行自动管理。虽说它给出了 convert\_layout 这类接口，可具体的内存排布逻辑对于开发者而言是半透明的类似“黑盒”的存在。在 NVIDIA 架构环境下，这套启发式策略能够良好运行，不过当转移到海光 DCU 或者其他国产 AI 芯片上时，鉴于这些硬件有着不同的 Cache Line 大小、Bank 数量、冲突模式，Triton 的隐式策略常常没办法生成最优的无冲突访问模式。
开发者很难借助位运算清晰地控制每一个数据的物理存储位置，致使性能在异构平台上出现比较严重的衰退情况。而在调度策略方面，Triton 的不少编译器优化 Pass 是依据 NVIDIA 硬件特性进行深度定制的。在移植到其他架构时，通常需要重新编写大量后端逻辑。

目前来看，现有编译方案在适配 Transformer 大模型、异构硬件的时候，会遇到三大核心挑战。首先是通用循环抽象没办法有效利用张量核心，而高层块级 DSL 又缺少对内存层次结构的显式控制。其次是现有模式过度依赖 CUDA 生态，结果在多样化的硬件后端很难实现统一的高性能。最后是算子融合优化导致调度搜索空间呈现组合爆炸，缺少领域知识引导的搜索很难收敛到高效解。基于这些挑战，本研究要建立一个具备表达力、可移植性、高效搜索能力的编译框架。

\subsection{硬件厂商高性能库的依赖与挑战}
% 在如 PyTorch、TensorFlow\cite{abadi2016tensorflow} 等深度学习框架的底层实现中，硬件厂商提供的高性能算子库长期扮演着基石的角色。为了在激烈的硬件竞赛中通过基准测试，GPU 厂商以及国产芯片厂商均投入了庞大的工程团队，针对特定的指令集架构和微架构特性，手工编写并调优核心数学库。这些库以 NVIDIA 的 cuBLAS\cite{cublas_doc}、cuDNN\cite{cudnn_doc} 以及 AMD 的 rocBLAS\cite{rocblas_doc}、MIOpen\cite{miopen_doc} 为代表，代表了当前通用矩阵乘与卷积（Convolution）运算的性能“天花板”。其高效性源于对底层硬件的极致压榨：一方面，库开发者通过汇编级微调，直接使用底层汇编语言编写内核，以精确控制寄存器分配、指令流水线排布以及指令级并行\cite{tanner2018tensile}。另一方面，库中包含成百上千个针对不同代际 GPU 架构和特定矩阵尺寸优化的内核变体，运行时系统会根据输入形状通过启发式算法动态选择最优实现。因此，在标准的矩阵乘法测试中，任何自动生成的代码想要超越这些经过数十年积累的手工库都极具挑战。
在 PyTorch、TensorFlow\cite{abadi2016tensorflow} 等深度学习框架的底层实现环节里，硬件厂商所提供的高性能算子库起着基础性的关键作用。为了能在竞争激烈的硬件比拼当中通过基准测试，GPU 厂商、国产芯片厂商都投入了规模庞大的工程团队，依据特定的指令集架构、微架构特性，亲自手工编写并且进行优化核心数学库。这些库以 NVIDIA 的 cuBLAS\cite{cublas_doc}、cuDNN\cite{cudnn_doc}、 AMD 的 rocBLAS\cite{rocblas_doc}、MIOpen\cite{miopen_doc} 作为典型代表，展现出了当下通用矩阵乘与卷积（Convolution, Conv）运算的性能所能达到的最高水平。它们的高效之处在于对底层硬件进行了非常充分的利用：一方面，库开发者借助汇编级别的细致调整，直接采用底层汇编语言来编写内核\cite{tanner2018tensile}，从而能够精准地把控寄存器分配、指令流水线的排列、指令级并行。另一方面，库里面包括了成百上千个针对不同代际 GPU 架构、特定矩阵尺寸进行优化的算子变体，运行时系统会依据输入形状运用启发式算法动态挑选最优的实现方式。所以，在标准的矩阵乘法测试中，任何自动生成的代码要想超过这些经过数十年积累而成的手工库都是非常具有挑战性的。

\section{论文主要研究内容}
针对 Transformer 类大模型在异构硬件部署中面临的适配复杂、手工优化成本高以及现有编译器表现不佳等挑战，本文设计了基于 MLIR 的 DeepGen 编译框架，实现高性能算子的跨平台自动生成，主要研究内容包括以下四个方面：

1. 面向异构硬件特性的块级分层中间表示设计与降级机制研究

% 为了解决传统深度学习编译器在代码降级过程中丢失算子块级语义的问题，本文基于 MLIR 基础设施，设计了一套专用的 DeepGen Dialect。
% 块级抽象建模：该 IR 体系采用分层设计思想，首先在线程块级别对矩阵计算进行粗粒度建模，保留了矩阵乘法与注意力机制的高维张量语义。其次，通过显式定义的 Alloc\_Buffer、Copy 与 Block、Matmul 操作，实现了计算逻辑与数据搬运的彻底解耦。这种解耦设计使得编译器能够独立地对访存行为进行精细化控制，为消除异构硬件上的存储访问冲突奠定了基础。
% 自动化降级推断：针对从线程块级到线程级的映射难题，本文提出了一种基于数据依赖关系的索引推断算法。该算法能够根据算子的计算模式，自动推导线程块与线程束内的细粒度索引布局，实现了从高层抽象到底层 SIMT（单指令多线程）执行模型的自动化且语义正确的转换，有效屏蔽了底层硬件指令集的差异。
为应对传统深度学习编译器于代码降级进程中出现的丢失算子块级语义这一状况，本文依托 MLIR 基础设施，设计出一套专用方言——DeepGen。块级抽象建模方面，此 IR 模式运用分层设计理念，先在线程块方面针对矩阵计算进行粗粒度建模，将矩阵乘法与注意力机制的高维张量语义予以保留。接着，借助显式定义的 Alloc Buffer、Copy、Block、Matmul 操作，达成了计算逻辑与数据搬运的完全解耦。这种解耦设计让编译器能够单独对访存行为实施精细化管控，为消除异构硬件上的存储访问冲突创造了条件。自动化降级推断方面，针对从线程块级到线程级的映射难题，本文给出一种基于数据依赖关系的索引推断算法。该算法可按照算子的计算模式，自动推导线程块与线程束内的细粒度索引布局，达成从高层抽象至底层 SIMT（单指令多线程）执行模型的自动化且语义无误的转换，有效屏蔽了底层硬件指令集的差别。

2. 异构平台下通用高性能编译优化 Pass 集合设计

% 针对 Transformer 关键算子在不同硬件上性能差距，本文设计了一套通用的、可插拔的编译优化 Pass 集合，涵盖了计算调度与访存管理的各个层面。
% 访存流水线优化：为了掩盖全局显存的高延迟，设计了软件流水线（Software Pipelining）优化 Pass，通过多级缓冲技术实现数据的预取，使计算指令与访存指令在时间上深度重叠。
% 并行度增强优化：针对 MHA 或者 GEMM 算子中规约维度过大导致线程块并行度不足的问题，提出了局部规约轴拆分（Local Split K） 优化 Pass。该技术将规约维度的计算任务切分至多个逻辑线程组，并在片上共享内存中进行中间结果的高效聚合，显著提升了硬件占用率。
% 访存冲突消除优化：针对海光 DCU 等硬件架构中特殊的 Bank 冲突问题，提出了离散化访存（Access Scatter） 与线程块重排（Block Swizzle）优化 Pass。通过对访存索引进行数学重写与离散化映射，显式地打散了连续地址的访问模式，在不改变计算语义的前提下彻底消除了共享内存的 Bank 冲突，同时大幅提升了 Cache 的命中率与有效带宽。
鉴于 Transformer 关键算子在不同硬件上呈现出来性能差异的状况，本文构思了一套具备通用性且能够插拔的编译优化 Pass 集合，这套集合于计算调度、访存管理的各个方面均有所涉及。访存流水线优化方面，为对全局显存的高延迟加以掩盖，设计出了软件流水线（Software Pipelining）优化 Pass，借助多级缓冲技术达成数据的预取行为，以此促使计算指令与访存指令在时间维度上实现深度重叠。并行度增强优化方面，因为矩阵乘法与注意力机制里规约维度过大而致使线程块并行度不够这一问题，提出了局部规约轴拆分（Local Split K）优化 Pass。此项技术把规约维度的计算任务划分至多个逻辑线程组，并且在片上共享内存里对中间结果展开高效聚合，进而提高硬件占用率。访存冲突消除优化方面，面对不同硬件架构中存在的 Bank 冲突问题，提出了离散化访存（Access Scatter）与线程块重排（Block Swizzle）优化 Pass。通过针对访存索引施行离散化映射，明确地打散了连续地址的访问模式，在不改变计算语义的情形下消除了共享内存的 Bank 冲突，与此同时显著提高 Cache 的命中率。

3. 基于空间剪枝策略的算子自动调优框架构建

% 为了解决手动调整编译参数耗时费力且难以收敛至最优解的问题，本文构建了一个闭环的自动性能调优框架。
% 参数空间生成：根据算子特性，抽象出固定种类的可进行调优的超参数，各算子根据自身计算特点构建该超参数配置文件，调优参数生成模块根据硬件特性组合生成基础调优空间，为剪枝模块提供数据空间基础。
% 高效剪枝策略：为了应对参数组合爆炸带来的搜索效率低下的问题，设计了多重剪枝策略。通过静态分析剔除会导致寄存器溢出或资源超限的无效配置，大幅缩小了搜索范围。框架通过实际运行并测量候选配置的性能，最终筛选出适配当前硬件特性的最优参数组合，实现了超越厂商手工库性能的自动化代码生成。
针对手动调整编译参数时耗时费力且难以达到最优解的状况，本文搭建了一个闭环的自动性能调优框架。在参数空间生成方面，根据算子特性，抽象出固定种类的可进行调优的超参数，各算子根据自身计算特点构建该超参数配置文件，调优参数生成模块根据硬件特性组合生成基础调优空间，为剪枝模块提供数据空间基础。在高效剪枝策略，识别硬件的特性设计了多重剪枝策略，例如静态分析去除会造成寄存器溢出或资源超限的无效配置，以及分析索引是否会造成 Bank 冲突等。最后框架通过实际运行并测量候选配置的性能，最终挑选出适配当前硬件特性的最优参数组合，达成了比肩厂商手工库性能的自动化代码生成。

4. 跨平台性能评估与端到端推理验证

% 为了验证 DeepGen 框架的有效性与泛化能力，本文在 NVIDIA A100 GPU 与 海光 Z100/K100 DCU 两类典型的异构计算平台上进行了广泛的实验。
% 算子级性能测试：分别对比了 DeepGen 生成的算子与官方高性能库、主流编译器在不同矩阵规模下的吞吐量，并深度分析了两个实验的加速效果及其性能来源。
% 数值精度验证：通过与标准 PyTorch 实现进行对比，计算结果的L2范数的相对误差，证明了本文所提出的 IR 转换与优化策略在保持数值计算正确性方面的可靠性。
% 端到端模型验证：将生成的算子集成至 BERT、GPT-2 及 LLaMA2 等主流 Transformer 模型的推理流程中，验证了优化后的算子对模型整体推理延迟的提升效果。
为验证 DeepGen 框架有效性和泛化能力，本文于 NVIDIA A100/V100 GPU、海光 Z100/K100 DCU 这两类典型异构计算平台进行实验。
在算子级性能测试方面，本文采用了官方的 BLAS 库与 DeepGen 之间进行了多角度的性能，同时也深度对比分析了当前主流的编译器。
数值精度验证环节，本文采用计算结果 L2 范数相对误差，通过和标准 PyTorch 实现对比，以此证明本文提出的 IR 转换与优化策略在保持数值计算正确性上的可靠性。
端到端模型验证中，本文把生成的算子集成到 BERT、GPT-2、LLaMA2 等主流 Transformer 模型推理流程里，验证优化后算子对模型整体推理延迟的提高效果。
