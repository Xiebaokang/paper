\section{面向硬件层次结构的多级IR设计与转换}
在本节中，首先我们针对传统 GPU 的硬件层次结构，提出了线程块（Block）和线程（Thread）层级的算子建模方案，并以构建多头注意力机制为例进行详细阐述，其次我们提出了一种算法用于解决线程块和线程层级在 Lowering 过程中的索引推断的问题，最后从线程块和线程层级的两层 IR 代码的角度出发，通过示例详细阐述 IR 代码的 Lowering 过程。

%                      2.1 算子的计算建模
% =============================================================
\subsection{算子的计算建模}
现代 GPU 等异构加速硬件在其编程模型中通常呈现“线程块-线程”两级并行层级。通过 blockId 与 threadId 构成的张量访存索引表达式，系统能够将计算任务精确映射至对应的线程块及其内部线程。针对这种层次化的硬件特征，本文提出了一种解耦的计算建模策略：将算子的运算逻辑划分为“线程块级”与“线程级”两个层次，并在中间表示的构建中使用不同粒度的 Operation 进行刻画。

在线程块层级的逻辑建模中，张量访存的索引计算仅依赖于 BlockId。该层级主要聚焦在线程块层级的访存模式与计算调度，屏蔽了块内线程的具体任务分配细节。这阶段的索引计算与任务分配策略层次较高，所以其 IR 表达也更为简洁。为了能够满足描述算子在线程块级别的计算行为，本文自定义了 MLIR 方言——DeepGen Dialect。表 \ref{tab:table2} 详细列出了该方言中的核心 Operation 及其语义的描述，线程块层级的 IR 将由这些自定义操作符构建。
线程层级的建模则重点转向块内线程的细粒度任务分配与计算实现。该层级通过引入 ThreadId，将线程块级的 Operation 进一步 Lowering 为若干个更为低级语义的线程级 Operation，从而确保整个 IR 的逻辑完整性与可执行性。

以 Transformer 模型中的多头注意力机制为例，图 \ref{3_2} 展示了其计算过程在双层 IR 模型下的数据流向。图中蓝色区域代表算子在线程块层级的 Operation 表达，橙色区域则为对应 Operation 向线程层级细化的内部结构。各 Operation 采用圆角矩形标识，张量数据采用直角矩形标识；数据名称中的 GM、SM、Reg 分别对应全局内存、共享内存及寄存器。涉及到数据访存的 Operation，其下方均标注了相应的索引表达式。图中利用紫色矩形框表示线性循环结构，粉色方框表示非线性循环结构，循环体的迭代变量、上下界及步长均已显式标明。如图所示，MHA 的输入为 GM(Q)、GM(K) 与 GM(V)，输出为 GM(O)，各张量的形状由下方中括号标注（其中 b、h、s、d 分别代表批大小 Batch Size、注意力头数 Head Number、序列长度 Sequence Length 以及注意力头维度 Head Dimension）。红色虚线菱形箭头表示 Operation 之间的数据依赖关系，例如在图 \ref{3_2}(a) 中，ReduceOp 必须在 CopyOp 执行完毕后方可启动。

在线程块层级的运算建模中，Operation 访问数据时采用的索引变量只使用 BlockId。如图 \ref{3_2}(a) 所示，第一个 MatmulOp 操作所需的输入参数为共享内存的分块，它们由 CopyOp 从全局内存中所拷贝。其中 \{bz, by, bx, 0\} 与 \{bz, by, i, j\} 分别表示向全局缓存 GM(Q) 与 GM(K) 发起线程块级拷贝的索引表达（$i, j$ 为循环变量）。在计算Matmul时，为了减少共享内存的压力，我们对于 SM(K) 采用了切块加载再累加的策略（分块尺寸为 $[16, 32]$），使得 MatmulOp 在单次迭代中仅需针对 SM(Q)[j:j+16, 32] 与 SM(K)[16, 32] 的子分块进行计算。

在 DeepGen Dialect 的设计中，KernelOp 与 ParallelOp 用于封装算子的全局计算域，定义线程块网格分布及块内线程规模，自身不涉及具体的内存访问与算术运算。而 BlockOp 则承担了线程块级别的逐元素（Element-wise）计算与部分数据访存任务。如图 \ref{3_2}(a) 所示，表达式 $acc\_s = exp(acc\_s - m)$ 明确界定了 BlockOp 内部执行的标量/向量计算逻辑，从而在线程块级 IR 中将此类泛型计算进行了统一且高效的封装。

图 \ref{3_2}(b) 进一步展示了部分线程块级 Operation 向线程层级细化的建模过程。在此阶段，线程块级的 Operation 被展开为若干个细粒度的线程级 Operation。虚线指明了 MatmulOp 与 ReduceOp 在线程层级的内部实现细节。

线程层级下，寄存器需要在线程上进一步划分，该层级下的访存、计算等操作均需要引入 ThreadId 进一步细化索引计算。以图 \ref{3_2}(a) 中的 MatmulOp 为例，在降低到线程层级的IR表达后，其整体的输入输出参数仍与线程块层级的表达一致，即以 SM(Q) 和 SM(K) 为输入，Reg(acc\_s) 为输出。但其内部不再简单地使用 MatmulOp 笼统概括，而是给出了由 arith、affine 等细粒度 Operation 组成的具体计算逻辑，即：共享内存块被加载到寄存器 Reg(Q) 和 Reg(K) 中由线程块中每个线程执行外积计算再累加得到最终结果，并存回到 Reg(acc\_s) 中。其中，数据从共享内存拷贝进入寄存器时，索引 \{bk, map1(tid, i)\} 与 \{bk, map2(tid, i)\} 分别表示访问 SM(Q) 和 SM(K) 的索引表示，其中 tid 表示线程块中线程索引 ID，而 i 表示每个线程取数的循环变量，map 则为与 tid 与 i 相关的索引重映射函数。

为了更细粒度的表达线程层级的访存、计算等语义，故我们采用了 MLIR 中内置的 Affine、Arith 以及 Math 等方言。其中 Affine 方言中有循环变化和仿射映射相关的操作，例如循环结构（affine.for），数据加载/拷贝操作（affine.load/store）等，Arith 方言中具有基本的算数操作，例如加、减、乘、除操作（arith.addf/subf/mulf/divf）等，而 Math 方言中则具有数学运算操作，例如指数运算操作（math.expf）、对数元素操作（math.logf）等。以上这些操作支持我们表达线程层级的复杂计算语义，同时暴露更多优化细节，更好的支持后续的优化。

% 在线程块层级表现为整体寄存器张量块（Reg）的数据，在线程层级被精确切割为每个物理线程私有的数据分块。该层级下的访存与计算必须引入 ThreadId 以实现索引的细粒度推断。以图\ref{3_2}(a)中的 MatmulOp 为例，在降级（Lowering）至线程层级后，其外围输入输出接口（即以 SM(Q)、SM(K) 为输入，Reg(acc\_s) 为输出）保持一致。但其内部不再由单一节点概括，而是替换为基于 Arith、Affine 等细粒度 Operation 的微观指令流：数据首先从共享内存加载至寄存器 Reg(Q) 和 Reg(K)，随后由块内各线程执行外积运算并累加，最终存回 Reg(acc\_s)。此过程中，数据从共享内存载入寄存器的索引表达式变为 \{bk, map1(tid, i)\} 与 \{bk, map2(tid, i)\}，其中 tid 代表当前线程的索引 ID（ThreadId），i 为循环变量，map 则为实现索引重映射的转换函数。

% 为了准确表达线程层级更细粒度的访存与计算语义，本文直接复用了 MLIR 社区成熟的内置方言。具体而言：利用 Affine 方言描述仿射映射相关的控制流（如 affine.for）与内存存取（如 affine.load/store）；利用 Arith 方言执行基础标量算术运算（如加、减、乘、除对应的 arith.addf/subf/mulf/divf 等）；利用 Math 方言处理复杂的数学超越函数（如指数运算 math.expf、对数运算 math.logf 等）。这些基础设施的引入不仅保障了复杂计算在底层线程的正确映射，还充分暴露了执行细节，为后续挂载高级优化 Pass 提供了广阔的操作空间。

\subsection{IR降级信息推断算法}

在上一节中，本文将算子建模过程划分为线程块层级与线程层级两个阶段。对应到 MLIR 实现中，首先使用 DeepGen 方言对算子逻辑进行表达，其次通过 IR Lowering 过程将线程块层级的操作进一步降低为线程层级的操作。在该 Lowering 过程中，需要通过引入 ThreadId 参与索引计算，从而将线程块层级的访存操作具体映射到每个线程的执行行为。因此，在执行 IR Lowering 之前，必须明确线 BlockId 与 ThreadId 之间的索引映射关系。

然而，在线程块层级的 IR 表示中，并未显式包含线程层级的索引映射信息，因此无法直接完成线程级访存索引的生成。为解决该问题，需要结合算子的计算语义以及线程块层级 Operation 的结构特性，对线程级索引映射关系进行推断，从而为后续 IR Lowering 提供必要的指导信息。

\begin{figure}[htbp]
\centering
\includegraphics[width=0.75\textwidth]{3_3}
\caption{LowerInfo 结构体定义}\label{3_3}
\vspace{\baselineskip}
\end{figure}

如图 \ref{3_3} 所示，本文设计了一种名为 LowerInfo（Lowering Information）的结构体，用于记录缓冲区（Buffer）在不同执行层级上的访存索引映射关系。对于每一个涉及缓冲区访问的 Operation，系统都会为其操作的 Buffer 建立对应的 LowerInfo 结构。这里的 Buffer 在 MLIR 中以 MemRef 类型（type）的 Value 表示。LowerInfo 结构内部采用 MLIR 的 AffineMap 来描述不同层级的索引映射表达式，并通过设置索引变量的上界约束保证访存地址的合法性。

在 LowerInfo 结构中，除了线程块与线程层级的索引映射之外，还引入了线程束层级的索引表达，从而能够实现更细粒度的线程访问控制。此外，结构中还包含用于访存离散化的索引映射机制，通过对线程访问的数据块进行分散化访问，可以有效降低共享内存中的 Bank Conflict 问题。针对双层离散化策略的具体实现将在第 \ref{ch3_5} 节中进行详细介绍。最终，在 IR Lowering 阶段，对于共享内存而言，只需将线程块层级 Operation 中的 Block 级索引表达式与 LowerInfo 中记录的线程、线程束等层级的索引表达式进行组合，而对于寄存器而言只需要组合 LowerInfo 中线程内部的索引表达式，即可得到线程级别的最终访存索引。

通过对算子执行过程的分析可以发现，不同 Operation 之间对 Buffer 索引的依赖关系存在明显差异。例如，在图 \ref{3_2}(a) 中，CopyOp与MatmulOp之间的索引访问彼此独立。具体而言，CopyOp 对共享内存 SM(K) 进行写入时的索引分布，并不要求与 MatmulOp 从 SM(K) 读取数据时的索引保持一致。而另一方面，MatmulOp 与 ReduceOp 之间通过寄存器 Reg(acc\_s) 传递中间结果，因此两者在索引访问上必须保持一致。由于 MatmulOp 计算得到的部分结果分别存储在不同线程的寄存器中，因此在 ReduceOp 执行时，每个线程必须按照 MatmulOp 中数据写入 Reg(acc\_s) 时的索引方式进行读取，才能保证计算结果的正确性。

基于上述分析可以发现，仅寄存器的 LowerInfo 需要在多个 Operation 之间保持一致，并参与跨 Operation 的依赖关系推断；而对于共享内存或全局内存等其他类型 Buffer，其 LowerInfo 仅在当前 Operation 内部有效。当这些 Buffer 在后续 Operation 中再次被访问时，需要根据新的 Operation 语义重新推断其 LowerInfo。因此，本文提出了一种 LowerInfo 推断算法 \ref{alg1}，用于自动生成算子中各访存 Operation 对应的 Buffer 索引映射信息，从而指导 IR 的 Lowering 过程。根据推断过程中对前序信息依赖程度的不同，线程块层级的 Operation 可分为两类。

\begin{figure}[htbp]
\centering
\includegraphics[width=0.75\textwidth]{3_4}
\caption{MatmulOp 中 Buffer 的 LowerInfo 的独立推断}\label{3_4}
\vspace{\baselineskip}
\end{figure}



第一类为可独立推断型 Operation。这类 Operation 可以根据其自身的计算语义直接推导出相关 Buffer 的 LowerInfo。例如图 \ref{3_2}(a) 中的 MatmulOp，其输入 Buffer 的索引分布可以由输出 Buffer 的索引关系推导得到。如图 \ref{3_4} 所示为 MatmulOp 中 Buffer LowerInfo 的推断过程。其中 Reg(X) 表示寄存器层级的 Buffer，Tx 表示线程束内部线程的索引，不同颜色表示不同线程束负责计算 Buffer 中的不同区域。由于该矩阵乘法采用外积（Outer Product）的计算方式，因此输出寄存器 Reg(acc\_s) 的 LowerInfo 可以直接根据矩阵乘法的计算规则确定。在计算过程中还引入两个临时寄存器 Reg(Q) 与 Reg(K)，分别用于存储来自 SM(Q$^T$) 和 SM(K) 的数据。尽管这两个寄存器的 LowerInfo 未在图中显式展示，但其索引关系同样可以由 Reg(acc\_s) 的 LowerInfo 推导得到。因此，通过外积计算的结构特性即可进一步确定 SM(Q$^T$) 与 SM(K) 的 LowerInfo 信息。该推断过程对应算法 \ref{alg1} 中的第 8 至第 15 行。

\begin{algorithm}
\caption{LowerInfo 推断算法}\label{alg1}
\setstretch{1.5}
    \KwIn{kernel\_op - 块级别的 KernelOp 操作实例}
    \KwOut{成功返回 (\textbf{op\_bufs\_info\_map}, \textbf{frag\_buf\_info\_map})，失败返回 false}

    // 步骤 0：初始化数据结构 \\
    \textbf{need\_infer\_ops} $\gets$ $\emptyset$ \\
    \textbf{op\_bufs\_info\_map} $\gets$ $\emptyset$ \\
    \textbf{frag\_buf\_info\_map} $\gets$ $\emptyset$ \\

    // 步骤 1：收集需要推断的操作 \\
    \textbf{need\_infer\_ops} $\gets$ \{op $\in$ \textbf{kernel\_op} $|$ op 类型为 CopyOp、BlockOp、MatmulOp、FillOp 或 ReduceOp\} \\

    // 步骤 2：直接推断 \\
    \ForEach{op $\in$ \textbf{need\_infer\_ops}（按前序遍历顺序）}{
        \If{op 满足直接推断条件}{
            \textbf{infos} $\gets$ op.inferLowerInfo() \\
            \textbf{op\_bufs\_info\_map}[op] $\gets$ \textbf{infos} \\
            从 \textbf{infos} 中更新寄存器级 LowerInfo 至 \textbf{frag\_buf\_info\_map} \\
            \textbf{need\_infer\_ops} $\gets$ \textbf{need\_infer\_ops} $-$ \{op\}
        }
    }
    // 步骤 3：迭代式依赖推断 \\
    \For{$k \gets 1$ \KwTo $1000$}{
        \If{\textbf{need\_infer\_ops} = $\emptyset$}{
            \Return (\textbf{op\_bufs\_info\_map}, \textbf{frag\_buf\_info\_map})
        }
        
        \textbf{del\_ops} $\gets$ $\emptyset$ \\
        \ForEach{op $\in$ \textbf{need\_infer\_ops}}{
            \If{基于 \textbf{frag\_buf\_info\_map} 中的已有信息可推断 op}{
                \textbf{infos} $\gets$ 基于依赖关系推断(op, \textbf{frag\_buf\_info\_map}) \\
                \textbf{op\_bufs\_info\_map}[op] $\gets$ \textbf{infos} \\
                从 \textbf{infos} 中更新寄存器级 LowerInfo 至 \textbf{frag\_buf\_info\_map} \\
                \textbf{del\_ops} $\gets$ \textbf{del\_ops} $\cup$ \{op\}
            }
        }
        
        \textbf{need\_infer\_ops} $\gets$ \textbf{need\_infer\_ops} $-$ \textbf{del\_ops}
    }
    
    \Return false

与 MatmulOp 不同，ReduceOp 的 LowerInfo 推断依赖于前序 Operation 的寄存器索引信息。如果在未知 Reg(acc\_s) 的 LowerInfo 情况下执行推断，将无法直接确定 Reg(m) 的索引分布。这是因为寄存器 Buffer 在 GPU 执行模型中为线程私有资源，当同一寄存器 Buffer 被多个 Operation 访问时，其索引映射必须保持一致，否则将可能导致计算错误甚至访存越界等问题。

如图 \ref{3_10} 所示为 ReduceOp 中 Buffer 的 LowerInfo 打印结果，其中注释表示组合后的索引表达式。在推断 Reg(m) 的 LowerInfo 时，首先利用 ReduceOp 的 dim 属性确定该 Operation 沿行方向执行区间最大值规约操作。同时，根据 Reg(acc\_s) 的 LowerInfo 可以得知每个线程所持有的数据分块形状为 [2, 4]。在沿行方向执行规约后，每个线程最终仅保留长度为 2 的结果向量，因此 Reg(m) 的数据分块形状为 [2]。由此可以得到 Reg(m) 的 LowerInfo，其索引分布等价于 Reg(acc\_s) 在行方向上的映射拷贝。

如图 \ref{3_8} 所示为 ReduceOp Lowering 后的线程层级 IR。其中第一步对 Reg(m) 缓存进行了清空以操作保证数据的正确性，然后使用 Reg(acc\_s) 进行最大值规约计算，结果写入了 Reg(m) 中，其中索引直接使用 thread\_inner\_index 中存储的索引，之后根据 LowerInfo 中的布局信息，可以得知整个线程块在 dim=1 方向上线程束的数量以及线程数量，利用 GPU Dialect 提供的线程束级别的 Shuffle XOR 操作，实现在线程块布局中属于同一行的线程之间的规约操作，最后将结果存储在 Reg(m) 中。

% 矩阵乘法与注意力机制是 Transformer 推理阶段最核心的计算单元。尽管二者在结构形式上有所差异，但从计算模式与性能特性角度分析，其优化问题具有高度一致性。本节将从计算结构、访存行为、并行映射与规约特性等方面对两类算子进行系统分析，为后续优化 Pass 的设计提供理论依据。
Transformer 进行推理时，主要的计算单元包括矩阵乘法和注意力机制。该两样事物在结构形式方面存在差异，但是从计算模式和性能特性角度分析，优化问题表现出高度的一致性。本文主要在计算结构、访存行为、并行映射和规约特性等方面对两类算子进行分析，为后续优化设计提供理论依据。

图 \ref{3_11}(a) 表现了计算结构的角度。矩阵乘法表现出标准的三重循环规约方式。它拥有一个重要的规约维度，也就是 K 维。输出矩阵所包括的各个子块在空间上各自独立。输入数据则拥有较大的数据复用潜力。如图 \ref{3_11}(b) 所示，在理想情况下，每个输出子块的计算可以在片上存储中完成大部分规约操作，并在有需要时才访问全局内存。在程序进行运行的过程中，由于片上存储空间比较有限，数据需要从全局内存进行多次加载。如果分块策略或者线程映射方式设计不够合理，带宽会成为主要的性能瓶颈。所以，矩阵乘法的理论算术强度虽然比较高，但是它的实际性能经常会被访存效率所限制。

% 注意力机制在结构上可以分解为两个矩阵乘算子与一个逐行归一化过程的组合。首先计算查询矩阵与键矩阵的乘积生成相关性矩阵，随后对每一行进行缩放与 Softmax 归一化，最后将归一化结果与值矩阵相乘得到输出。从计算本质上看，MHA 的主要计算负载仍集中于矩阵乘阶段。然而，与单纯的 Matmul 相比，MHA 在中间阶段会生成规模随序列长度平方增长的相关性矩阵，这不仅增加了计算量，也带来了更为显著的访存压力。若中间结果频繁写回全局内存再读取，将显著降低整体执行效率。因此，MHA 的优化不仅需要关注计算调度问题，还必须考虑中间数据的存储与复用策略，如图 \ref{3_12} 所示，这里我们采用Safe softmax（如 \ref{softmax}）实现融合 MHA，与传统MHA相比，无需将 $QK^T$ 的结果写回 HBM 中从而降低访存压力。
注意力机制由两个矩阵乘算子和一个逐行归一化过程组合而成。首先通过查询矩阵和键矩阵相乘来计算相关性矩阵，然后对每一行进行缩放和 Softmax 归一化，最后把归一化结果与值矩阵相乘以得到输出。从计算本质来看，MHA 的主要计算负荷仍然集中在矩阵乘那个阶段。但是，和基础矩阵乘法计算相比，多头注意力机制进行中间处理时会生成相关性矩阵，其规模随序列长度平方增长，这种情况使计算负担增加，并且也使访存需求更加突出。如果中间结果经常被写回全局内存后又被读取，那么会明显降低整体的执行效率。MHA 的优化不只要考虑计算调度问题，还得注意中间数据的存储和复用方式，如图 \ref{3_12} 所示，本文使用 Safe softmax（如 \ref{softmax}）来实现 MHA 的融合，对比传统 MHA 方法，该方法不用把 $QK^T$ 的结果写回HBM，减少了访存压力。

\begin{equation}\label{softmax}
y_i = \frac{e^{x_i-\sideset{}{}\max _{k=1}^V x_k }}{ {\textstyle \sum_{j=1}^{V}e^{x_j-\sideset{}{}\max _{k=1}^V x_k }} } 
\end{equation}

% 从性能瓶颈角度分析，矩阵乘与 MHA 均呈现出“计算能力充足、带宽压力显著”的特征。根据 Roofline 模型，当数据复用不足时，算子执行将落入带宽受限区域。特别是在长序列或大规模隐藏维度场景下，全局内存访问次数迅速增加，若无法通过分块与共享内存机制提高数据复用率，则硬件计算单元难以达到高利用率。因此，提升数据在片上存储中的停留时间与复用程度，是优化的首要方向。

% 进一步分析线程块级执行行为可以发现，不同线程块之间往往访问输入矩阵中存在重叠的数据区域。若线程块按照简单的行主序遍历输出空间，则相邻块在时间上的访问局部性较弱，缓存命中率下降，导致重复加载相同数据。通过调整线程块的遍历顺序，使空间上邻近的计算块在时间上连续执行，可以有效提高缓存利用率并降低全局内存访问次数。这一问题本质上是线程块索引映射与调度顺序优化问题。
在性能瓶颈角度上，矩阵乘和 MHA 都表现为拥有充足计算能力时，带宽压力会变得非常明显。Roofline 模型表明，当数据复用程度较低时，算子执行会处于带宽受限区域。在长序列或大规模隐藏维度的条件下，全局内存访问次数会快速增加，如果不利用分块和共享内存方式提高数据复用率，硬件计算单元就很难保持较高的利用率。所以，增加数据在片上存储中的停留时间并且提高复用程度，属于优化的主要方向。对线程块的级执行行为进行分析可以看出，多个线程块经常会访问输入矩阵里有重叠的数据区域。线程块如果以简单的行主序遍历输出空间，相邻块的时间访问局部性比较差，缓存命中率会下降，系统因此需要重复加载相同数据。改变线程块遍次序能让空间上相邻的计算块在时间上进行依次执行，这会提高缓存利用效率并使全局内存访问频次减少。该问题主要出现在线程块索引映射和调度顺序的优化方面。

% 此外，矩阵乘中的规约维度通常由单个线程块内部完成，但当规约维度较大时，单块计算可能成为性能限制因素。从理论上讲，规约轴可以被划分为多个子段，由多个线程块并行计算部分结果，再通过额外的合并步骤完成最终输出。这种策略能够提高并行度，但同时也引入额外的归并开销，因此需要在并行效率与同步成本之间取得平衡。MHA 中的矩阵乘阶段同样具备类似特性，因此规约轴并行策略具有通用意义。
% 最后，计算与访存之间的执行关系也是影响性能的重要因素。在传统实现中，数据加载与计算往往串行进行，而全局内存访问延迟远高于算术操作延迟，导致计算单元在等待数据时处于空闲状态。若能够通过流水线机制将数据加载阶段与计算阶段进行重叠，则可以有效隐藏访存延迟，提高整体吞吐率。这种“访存—计算重叠”的思想对于矩阵乘与 MHA 均具有重要意义。
矩阵乘里的规约维度一般是在单个线程块内部完成的，不过当规约维度比较大时，单个线程块的计算可能会成为限制性能的因素。理论上，规约轴可以被分为几个子段，多个线程块并行计算部分结果，再通过额外的合并步骤来完成最终输出。该策略会让并行度有所提高，可是会带来额外的归并开销，因此需要在并行效率和同步成本之间寻求平衡。MHA 中矩阵乘阶段也具有类似特性，规约轴并行策略因此有通用意义。最后计算与访存之间的执行关系会也是影响性能的主要因素。在传统实现方式中，数据加载和计算通常以串行方式进行进行，而且全局内存访问的延迟要比算术操作的延迟高出很多，这会使计算单元在等待数据时处于空闲状态。如果使用流水线技术，让数据加载工作和计算工作一起进行，会遮挡访存延迟，增加整体吞吐量。访存和计算重叠的思想对矩阵乘、MHA 都有重要作用。

\subsection{Software Pipeline：软件流水线优化}
% 在深度学习算子中，循环结构通常呈现为“加载—计算—存储（load-compute-store）”的执行模式。在这种模式下，计算指令通常依赖于前序访存指令返回的数据，从而形成严格的数据依赖链。当访存延迟较高时，GPU 需要等待内存访问完成后才能继续执行计算操作，导致计算单元在等待期间处于空闲状态，严重限制算子性能的发挥。尽管现代 GPU 具备硬件指令流水线与一定程度的乱序发射能力，但跨指令的数据依赖仍会迫使访存与计算阶段串行执行，从而降低计算核心的利用率。
在深度学习算子里，循环结构通常表现为加载、计算和存储，也就是所谓的 load-compute-store 执行模式。在该模式中，计算指令大多依靠前序访存指令提供的数据，进而建立起严格的数据依赖链。在访存延迟比较高的情况下，GPU 需要等内存访问完成以后才可以继续进行计算，这会使计算单元在等待时处于空闲，严重限制了算子性能的发挥。现代 GPU 具备硬件指令流水线、部分乱序发射能力，跨指令数据依赖会使访存和计算阶段进行串行执行，并且使计算核心利用率出现下降。

% 为隐藏全局内存访问以及共享内存到寄存器之间的数据访问延迟，本文设计并实现了面向线程块与线程两个层级的双层软件流水线优化 Pass。该 Pass 首先通过数据流分析识别循环体内部“load-compute-store”结构中的依赖关系，然后通过多缓冲（MultiBuffer）与索引重写技术，将原本强依赖的访存与计算操作拆分为多个相互独立的流水阶段，自动构建由 Prologue、Main Loop 和 Epilogue 组成的标准三段式软件流水线结构。通过该变换，下一次迭代的数据加载可以与当前迭代的计算操作并行执行，从而有效隐藏访存延迟并提升指令级并行度。

为隐藏全局内存访问以及共享内存到寄存器之间的数据访问延迟，我们设计了针对线程块级和线程级的双层软件流水线优化 Pass。该 Pass 首先通过数据流分析识别循环体内 load-compute-store 三阶段结构中严格的前后依赖关系，随后通过双或多缓冲索引重写，将原本单次迭代访存与计算解耦为相互独立的流水阶段，自动构造标准的 Prologue、Main Loop 和 Epilogue 三段式软件流水线。变换后，下一迭代的数据加载可与当前迭代的计算完全重叠，实现访存延迟的有效隐藏，显著提升指令级并行度与实际算子吞吐。

% 在具体实现过程中，原始循环结构被拆分为 Prologue、Main Loop 和 Epilogue 三个阶段。其中，Prologue 阶段负责完成初始数据预取；Main Loop 阶段为流水线的稳态执行阶段，在每次循环迭代中同时执行访存与计算操作；Epilogue 阶段则用于完成最后一轮尚未执行的计算任务。在 Main Loop 中，循环起始位置被调整为原循环的第二次迭代，而循环上界保持不变。由于访存操作与计算操作使用不同的索引偏移，因此单次迭代中的访存与计算不再存在直接依赖关系，可以实现并行执行，从而有效提升 GPU 执行单元的利用率。

软件流水线优化的原理主要是消除访存操作和计算操作之间存在着的直接依赖关系，也就是让计算阶段的输入来源于前一阶段已完成加载的数据，同时当前阶段会为正在进行的下一次迭代准备数据。采用这种方式，在进行第 $n$ 次迭代计算时，系统可以同时载入第 $n+1$ 次迭代所需数据，这样便实现了存取操作和计算操作的重叠执行。在具体进行实现的过程中，原始的循环结构被拆分成 Prologue、Main Loop 和 Epilogue 这三个阶段。其中，Prologue 阶段负责完成初始数据预取。Main Loop 阶段是流水线在稳态进行执行的过程中，每次循环迭代都会进行访存操作和计算操作。Epilogue 阶段主要用来进行最后一轮尚未完成的计算任务。主循环将循环起始位置调整到原循环的第二次迭代，循环上界保存不变。访存操作和计算操作使用各不相同的索引偏移，单次迭代中的访存和计算便不会有直接依赖关系，可以并行执行，进而提高 GPU 执行单元的利用率。

针对上述问题，本文设计并实现了 LocalSplitK Pass，通过对规约轴方向的计算任务进行重新划分来提升并行度。其核心思想是将原本由单个线程块完成的规约计算在逻辑上拆分为多个子任务，并在同一线程块内并行执行。如图 \ref{3_19}(c) 所示，原本由一个线程块计算的输出区域被划分为多个独立子区域，同时线程块内部的线程数量按比例扩展。每个线程组负责计算规约轴上不同子区间的部分结果，并将中间结果暂存于共享内存中。计算完成后，通过一次同步操作对共享内存中的部分和进行规约累加，从而得到最终输出结果。

\subsection{Block Swizzle：线程块访存重排优化}

% 在异构计算体系中，缓解“内存墙”问题是提升整体计算性能的重要目标。尽管软件管理的共享内存能够为规则数据访问提供高效的显式局部性管理，但在复杂访存模式下，硬件管理的通用缓存（如 L1/L2 缓存）仍然发挥着不可替代的作用。通用缓存能够自动聚合线程束内的非合并访问请求，从而提升带宽利用率，并为指令流和随机访存提供透明的局部性保障。在 Transformer 推理等典型应用场景中，当连续调度的线程块访问的全局内存区域存在较高重叠度时，可以显著提升 L2 缓存的复用率，从而减少对全局内存的访问压力。
% 基于上述观察，本文实现了 BlockSwizzle Pass，通过对线程块 ID 进行轻量级重映射来优化线程块的调度顺序。该方法通过建立逻辑 blockId 与物理 blockId 之间的映射关系，打破默认的行优先分配模式，使访问相邻数据区域的线程块在时间上更加接近，从而增强线程块之间的时空局部性，提高缓存复用率。
在异构计算系统中，减轻内存墙问题是提高整体计算性能的主要目标。软件管理的共享内存可以进行高效显式局部性管理来控制规则数据访问，但在复杂访存模式中，硬件管理的通用缓存，如 L1 或者 L2 缓存，仍然有不可替代的作用。通用缓存可以把线程束内的非合并访问请求自动进行聚合，使带宽利用率得到提高，并为指令流和随机访存提供透明的局部性保障。在 Transformer 推理等多种应用场景中，连续调度的线程块访问的全局内存区域重叠度较高时，L2 缓存复用率会得以提高，由此可以减轻全局内存访问的压力。根据上述观察内容，本文实现了 BlockSwizzle Pass，通过对线程块 ID 进行轻量级的重映射，来优化线程块的调度顺序。该方法建立了逻辑 blockId 和物理 blockId 之间的映射关系，把行优先分配模式进行改变，让访问相邻数据区域的线程块在时间上更加靠近，从而改善了线程块之间的时空局部性，达到缓存复用率提高的目的。

图 \ref{3_23} 以矩阵乘法计算为例展示了 Block Swizzle 的优化原理。图中 A 和 B 为输入矩阵，C 为输出矩阵，每个小方格表示一个线程块负责计算的输出区域，因此计算矩阵 C 需要启动 64 个线程块。如图 \ref{3_23}(a) 所示，在默认的行优先执行模式下，线程块按照行主序依次计算矩阵 C，我们将需要加载A的一行数据和整个 B 矩阵的数据，也就是 72 个数据分块。
该计算模式下，缓存中的数据将很快会被刷新，当下次需要用到相同数据时，只能从全局内存中重新加载。采用 BlockSwizzle Pass 优化后，连续的四个线程块负责计算如图 \ref{3_23}(b) 矩阵中的阴影所示区域。此场景下我们只需要读 取 48 个数据分块，且该计算模式下，线程块访存 A 和 B 矩阵的数据重叠性更高，缓存中数据刷新速率较低，因此可有效提高缓存命中率。

\subsection{Access Scatter：离散化访存优化}\label{ch3_5}

% 在 GPU 体系结构中，共享内存通常被组织为 32 个 Bank，每个 Bank 宽度为 4 字节。当一个线程束中的线程请求的数据量超过 4B 时，硬件会将其拆分为多个内存事务，每个事务的最大传输规模为 128B。Bank 冲突的检测以单个事务为单位：当同一事务中的多个线程访问同一 Bank 的不同地址时，即会产生 Bank 冲突，从而导致访存请求被串行化处理。冲突路数越多，有效带宽越低，对算子性能的影响也越显著。因此，在 GPU 算子优化过程中，通过合理设计数据布局与访存模式以避免事务内的 Bank 冲突具有重要意义。
GPU 体系结构将共享内存都被组织为 32 个 Bank，每个 Bank 的宽度为 4 字节。若一个线程束中每个线程请求的数据总量超过 4B，硬件会将其拆分为多个内存事务，每个内存事务为 128B。Bank 冲突的计算以每个事务为单位，即当同一事务内的线程访问同一 Bank 的不同地址时，会发生 Bank 冲突，导致串行访问。冲突路数越多，有效带宽越低，性能下降越严重。因此，优化数据布局以规避事务内的 Bank 冲突至关重要。

\vspace{\baselineskip}
\end{figure}

以图 \ref{3_26} 与图 \ref{3_27} 为例，展示了图 \ref{3_2} 中第二个 MatmulOp 操作在不同访存策略下的计算示意。该算子的输入参数 Reg(acc\_s) 与 SM(V) 的形状分别为 [32, 32] 和 [32, 128]，分别位于寄存器与共享内存中。由于 Reg(acc\_s) 位于寄存器中，其线程级数据布局已确定，因此 SM(V) 与结果 Reg(acc\_o) 的 LowerInfo 需要根据 Reg(acc\_s) 的布局进行推导。在该配置下，每个线程持有 Reg(acc\_s) 中的 [2,4] 数据块，因此需要从 SM(V) 中加载大小为 [2,16] 的数据块以完成外积计算。

如图 \ref{3_26} 所示，线程 T0 需要从 SM(V) 中加载一个 [2,16] 的数据块。由于线程最大向量化加载宽度为 16 字节，每次最多读取 4 个元素，因此该数据块需要通过 4 次迭代完成加载。在每次迭代中，线程需要加载一行 16 个元素，而每行加载通过 4 条 LDS.128 指令完成，对应图中不同灰度标识的数据块。根据 SM(V) 在共享内存 Bank 上的映射关系可以观察到，在同一迭代中，0-7 号线程会访问同一 Bank 的不同地址，因此产生 4 路 Bank 冲突，从而降低访存效率。

该优化 Pass 通过修改线程级别的访存模式实现，也就是修改 LowerInfo 中与离散化相关的参数，若 xxx\_scatter\_bound 参数中上界全为 1，则证明没有设置访存离散化优化，因此线程的访存索引将不会被进行分散。因此我们通过缩小原本线程块、线程束和线程访问的数据分块大小，并通过使用相同的索引方式进行重复访存，其中每次重复的偏移由 LowerInfo 中 xxx\_scatter\_index 变量存储的索引表达式进行计算，最后使得线程每次访存的地址都不同，从而实现访存离散化优化。

% 该优化Pass主要通过调整线程层级的访存模式实现，即修改 LowerInfo 结构中与离散化相关的参数。当 xxx\_scatter\_bound 参数的上界均为 1 时，表示未启用访存离散化优化，此时线程按照原始布局进行连续访问。启用离散化后，系统会缩小线程块、线程束与线程在共享内存中的访问分块尺寸，并通过重复访问的方式重组数据加载顺序。每次重复访问的偏移量由 LowerInfo 中的 xxx\_scatter\_index 索引表达式计算得到，从而确保线程在不同访问阶段访问到不同的内存地址。

综上所述，AccessScatter Pass 通过在 IR 层引入离散化访存索引机制，对线程级共享内存访问模式进行重排，从而有效降低甚至消除 Bank 冲突问题。该方法在不改变计算语义的前提下显著提升共享内存带宽利用率，并为不同算子配置提供灵活的访存布局选择，为后续自动调优过程提供了更多潜在的高性能实现方案。

\section{基于空间剪枝的自动调优框架设计}
本文在前面所述的 IR 优化模式上，形成了一种利用空间剪枝进行自动调优的模式。该模式在 IR 层定义结构化调优参数模式，对算子执行结构进行参数化描述，并在此基础上形成可以搜索的调优空间。本文使用若干同硬件资源约束和算子结构特性相关的剪枝策略，在调优阶段事先剔掉不合理或低效的参数组合，把搜索空间缩小，并提高调优效率。自动进行编译、运行和性能评估，该模式可以在合理时间内为目标算子生成高性能的实现。本文先介绍自动调优模式的模式的整体设计和工作流程，接着给出调优空间参数的定义及其生成方式，最后对调优空间的剪枝策略做详细说明。

%                       4.1 调优框架的设计
% =============================================================
\subsection{调优框架的设计}

% 在算子代码生成过程中，多个关键调优参数共同决定了中间表示的变换路径及最终生成代码的性能表现。这些参数包括但不限于线程块与线程级的数据分块尺寸、软件流水线阶段设置、局部规约轴的切分策略，以及线程束、线程块与线程之间的布局结构等。它们共同构成了一个高维、离散且高度非凸的联合优化空间。

\subsection{搜索空间剪枝策略}
在自动调优过程中，算子的性能一般会由多个调优参数来决定。本文以线程块级数据分块尺寸、线程级计算粒度、规轴切分策略、软件流水线阶段数、线程块映射方式等参数为例，进行分析和研究。各参数互相之间存在比较复杂的耦合关系，不同参数的搭配组合对应着完全不同的执行策略。从理论上看，对所有可能的参数组合进行穷举搜索，可以得到最优的实现方案。不过，实际应用时，调优参数一般会形成一个高维离散空间，该空间的大小会随着参数数量的增多，以指数级的方式变大。如果直接对全部候选方案进行编译和测试，则会产生很高的时间开销和计算资源消耗，会对自动调优的效率产生影响。

\centering
\includegraphics[width=0.95\textwidth]{3_34}

\vspace{\baselineskip}

% 为了实现这一目标，本文在调优空间生成之后引入了一套分层剪枝机制。该机制通过对参数组合进行静态分析，从多个维度对候选配置进行筛选，在保证算子功能正确性的前提下尽可能减少需要实际编译与测试的方案数量。整体剪枝流程如图 \ref{3_34} 所示，其核心思想是将剪枝策略划分为两类：一类为基于硬件资源约束与架构特性的必要剪枝条件，另一类为基于专家经验与性能启发的可选剪枝条件。通过这两类策略的协同作用，可以在保证搜索空间覆盖潜在高性能方案的同时有效降低搜索复杂度。

% 在完成初始调优空间构建后，搜索空间中往往仍包含大量可执行但性能潜力较低的配置，我们首先引入一系列基于经验规则的可选剪枝策略，对明显不具备良好性能潜力的参数组合进行提前过滤。例如，在数据拷贝阶段，为保证访存效率，通常需要保证每个线程承担一定规模的数据搬运任务。当线程数据拷贝量过小时，线程调度与同步开销将显著增加，从而影响整体执行效率。因此，在调优空间生成过程中可以设定线程数据拷贝的最小负载下限，以避免生成过于细粒度的配置方案。此外，在矩阵乘算子中，分块形状（tile shape）对数据复用具有重要影响。实践经验表明，当输出 tile 的形状接近方形时，通常能够在行列两个方向获得更均衡的数据复用。因此，在搜索空间规模过大时，可优先保留接近方形的数据分块组合，从而减少需要探索的参数规模。

% 在完成可选剪枝后，调优框架进一步执行必要剪枝策略，以保证所有候选方案在目标硬件上均能够正确执行。该类剪枝主要基于 GPU 的硬件资源限制与执行模型约束。例如，每个线程块可使用的共享内存容量与寄存器数量均存在上限，当某些参数组合导致资源需求超出硬件限制时，该方案将无法编译或运行，因此需要在调优阶段提前剔除。此外，线程块组织方式还需满足线程束（warp）的结构要求，例如线程数量通常应为线程束大小的整数倍，否则会降低硬件利用率。同时，在共享内存访问过程中，不合理的线程映射还可能引发严重的 Bank Conflict，从而降低访存带宽利用率。因此，在必要剪枝阶段需要统一检查这些硬件约束，以过滤掉无法执行或明显低效的配置组合。

% 通过上述两类剪枝策略的协同作用，原始调优空间能够得到显著压缩。一方面，必要剪枝条件保证了所有候选方案在硬件资源与执行模型上均具有可行性；另一方面，可选剪枝条件则利用经验规则进一步缩小搜索范围，使自动调优过程能够更加集中地探索具有潜在高性能的配置区域。在实际应用中，当调优空间规模较小时，可以仅启用必要剪枝以保证搜索的完整性。而当调优空间规模过大时，则可以同时启用可选剪枝策略，从而显著降低搜索复杂度并加快调优收敛速度。通过这一机制，本文所提出的自动调优框架能够在保持较高搜索效率的同时，仍然有效发现性能优良的算子实现方案。

为了提高调优效率，本文在生成调优空间之后引入一种分层剪枝方式，该方式在保证算子功能正确和硬件可以执行的基础上，尽可能地减少实际需要进行编译和测试的候选方案数量。整个流程见图 \ref{3_34}。剪枝策略按照作用方式的区别可以分成两类，包括依靠硬件资源约束和架构特性的必要剪枝、依靠经验规则和性能启发的可选剪枝。前者用来保证方案可行，后者用来进一步压缩搜索空间。

初始调优空间生成完成以后，所有参数组合虽然都符合基本的生成条件，但是其中仍然存在大量可以在运行环境使用但性能潜力偏低的配置。本文首先运用可选剪枝策略，将明显不具备优势的方案提前进行过滤。例如在数据复制过程中，每个线程为了保证访存效率，一般会负责进行一定规模的数据搬运工作。如果拷贝的数据量很少且只使用单线程，线程调度、同步的开销就会增大，进而影响执行效率。矩阵乘算子里，分块形状会影响到数据复用的效果。输出了面 tile 接近正方形时，行列方向的数据复用更加均衡，在搜索空间较大的时候，会优先保留该分块组合。

在完成可选剪枝后，调优模式接着进行必要剪枝，让剩余方案可以在目标硬件上正确运行。本部分主要根据 GPU 的资源限制和执行模型来进行展开。共享内存和寄存器资源都有上限，当某组参数让资源需求超过硬件限制时，需要进行剔除。线程块的组织方式也需要满足 Warp 的基本要求，不这样会影响硬件利用率。如果共享内存访问方式不合理，会导致 Bank 冲突，会让访存带宽利用率下降。

把前面提到的两种剪枝策略结合起来进行使用，原始调优空间可以得到有效的压缩，调优过程主要分布在潜在的高性能区域中。当搜索空间较小时，系统可以只启用必要的剪枝操作，而在搜索空间较大时，系统可以同时启用可选剪枝操作，这样可以减少搜索复杂度，使调优问题的效率得到提高。

\section{本章小结}
% 本章详细阐述了面向 Transformer 模型关键算子的代码生成与 DeepGen 的核心设计与实现细节。针对现有编译器在块级逻辑表达和跨平台性能迁移方面的局限性，本章从中间表示构建、编译优化变换以及自动化调优三个维度提出了一套系统化的解决方案。
本章主要说明面向 Transformer 模型主要算子的代码生成，并对 DeepGen 的中心设计与实现细节进行介绍。针对现有编译器在块级逻辑表达和跨平台性能迁移上的局限，本章从中间表示建立、编译优化变换和自动化调优三个方面形成了一套整体的解决方案。

本章提出了一种面向异构硬件层次结构的多级中间表示建模方法。本文设计了名为 DeepGen 的方言，把算子的计算逻辑在线程块级和线程级两个层次上进行解耦。本文设计了利用寄存器依赖推断降级信息（LowerInfo）的算法，并且解决了将粗粒度块级表示转换为细粒度线程级表示的索引映射问题，可以准确地描述 GPU 计算调度和访存路径。

在分析矩阵乘法和注意力机制的算术及访存特性后，本章设计并实现了一组针对性的高性能优化 Pass 集合。具体内容为通过软件流水线来隐藏全局和共享内存在访问时产生的延迟。把局部规约轴进行分割可以增加长规约轴情况下的线程级并行度，并且通过线程块访存重排方式来提高 L2 缓存复用率。引入离散化访存可以消除共享内存的 Bank 冲突。多个 Pass 协同工作，显著突破了算子执行的内存与计算瓶颈。

为了解决多维优化参数组合爆炸问题，本文形成了利用空间剪枝的自动调优模式。本文通过定义结构化的调优参数模式，并结合硬件资源约束下的必要剪枝和专家经验指导的可选剪枝策略，剔除了无效、低效的参数组合。从而提高模型的运行效率。该模式实现了编译、测试和监控的异步协同，在很大程度缩小搜索空间的同时，确保调优的高效性。
