\section{异构计算加速平台}
\subsection{异构计算概述}
随着深度学习技术的爆发，特别是基于Transformer的大语言模型参数量达到万亿规模，传统以CPU为核心的同构计算体系已无法满足AI任务对计算和访存的需求，面向人工智能的异构计算因此成为支撑现代深度学习模型训练与推理的基础设施范式。深度神经网络和Transformer模型的高度规则性和数据并行性，促使计算架构从“以控制为中心”的通用CPU转向“以数据为中心”的异构协同模式，其中CPU负责逻辑控制、数据预处理与调度，而加速器则专注执行矩阵乘法、卷积及注意力等核心算子。

为争夺大模型时代的算力主导权，全球半导体厂商推出了多种技术路线的异构加速平台，形成了多样化但生态割裂的局面。主流平台主要包括通用图形处理器（GPGPU）路线和专用人工智能芯片（ASIC）路线。GPGPU以NVIDIA GPU为代表，凭借其CUDA生态成为当前最主流和成熟的硬件载体。AMD GPU及国产的海光DCU也采用类似架构，是构建国产自主算力底座的关键力量。ASIC路线则包括华为昇腾NPU、Google TPU和寒武纪MLU等，针对深度学习计算进行定制化设计，以追求极致能效比。
然而，不同平台在指令集架构、微架构设计、存储层级和编程模型上存在显著差异，这导致上层深度学习框架难以通过统一后端充分发挥各类硬件的性能。在众多方案中，NVIDIA GPU代表了工业界的性能标杆与优化基准，而海光DCU则体现了国产化替代的主流方向与典型架构。深入理解这两类代表性硬件的架构及编程模型，是分析算子性能瓶颈、设计跨平台编译优化策略的前提。

随着深度学习领域的不断发展，NVIDIA GPU凭借其卓越的并行计算能力与成熟的CUDA软件生态，已成为模型训练与推理的事实标准平台。其硬件架构历经Volta、Ampere至最新的Hopper、Blackwell等架构迭代，持续引入针对AI负载的专用优化特性，如图~\ref{2_1}~展示了配备 144 个 SM 的完整 NVIDIA H100 GPU 核心。

在硬件架构层面，GPU的核心计算单元是流多处理器（Streaming Multiprocessor, SM），如图~\ref{2_2}~所示为 NVNDIA H100 GPU 的 SM 架构图，其中每个SM包含大量的CUDA Core用于标量运算，以及针对深度学习定制的Tensor Core。Tensor Core作为专用加速单元，能够高效执行混合精度的矩阵乘累加运算，是Transformer模型中GEMM与Attention算子的核心加速引擎。为缓解内存瓶颈，GPU采用显式的多级存储体系，包括高速但容量有限的寄存器堆、可作为可编程暂存空间的共享内存（L1缓存），以及高容量但高延迟的全局显存（HBM）。高效利用这一存储层次，尤其是通过分块技术优化共享内存的访问，是实现高性能计算的关键。
\begin{figure}[htbp]
\centering

在编程模型层面，CUDA采用单指令多线程（SIMT）模型，通过大规模线程并行来掩盖访存延迟。如图~\ref{2_3}~所示,在 NVIDIA GPU 编程模型中，其线程组织呈现为“网格-线程块-线程”的三级逻辑抽象，其中线程簇（Cluster）层级为 Hopper 架构后特有抽象层级，不属于传统编程模型的层次结构。在同一个Cluster的Block可进行数据通信，减少对HBM的访问。硬件执行的基本单位是线程束（Warp），由32个线程组成。同一Warp内的线程以锁步方式执行相同指令，若出现条件分支会导致分支发散，严重损害执行效率。线程块是资源分配与调度的基本单位，被映射到SM上执行。SM内的Warp调度器通过动态上下文切换机制，在等待数据的Warp和就绪计算的Warp之间快速切换，以实现计算与访存的重叠，从而隐藏延迟。因此，维持足够的占用率，即在SM上驻留充足的活跃Warp以保持计算流水线持续饱和，是发挥GPU极致性能的核心原则。

当 Kernel 启动时，全局调度器将 Grid 中的线程块分配给各个 SM。在 SM 内部，Warp 调度器负责从处于就绪状态的 Warp 中选取指令进行发射。这种设计通过上下文切换机制实现了延迟隐藏：当某个 Warp 正在等待全局显存的数据时，调度器会立即切换到另一个已有数据准备好的 Warp 进行计算。因此，为了充分利用 GPU 的算力，必须保持足够高的占用率，即在 SM 上驻留足够多的活跃 Warp，以确保计算流水线永不停歇。总之，深入理解NVIDIA GPU从SM微架构、多级存储到SIMT执行模型的软硬件协同设计原理，是进行底层算子性能分析与优化不可或缺的基础。

\subsection{海光DCU架构及编程模型}
海光DCU是国产异构计算领域的代表性硬件平台，广泛应用于高性能计算中心与大规模AI智算集群。其设计理念继承了GPGPU的大规模并行架构，采用与NVIDIA类似的SIMT（单指令多线程）执行模型，但在微架构实现与资源组织方式上存在显著差异，这些差异成为跨平台算子优化的关键考量因素。
\begin{figure}[htbp]
\centering
\includegraphics[width=0.95\textwidth]{2_4}
\caption{DCU Z100 (CNDA1.0) CU 架构}\label{2_4}
\vspace{\baselineskip}
\end{figure}

在硬件架构方面，海光DCU获得了AMD的CDNA架构的技术授权，因此其架构特征与AMD CDNA架构基本相似，其核心计算模块是计算单元（Compute Unit, CU），功能上对应于NVIDIA的SM。如图~\ref{2_4}~为其CU的架构图，其中每个CU内部集成了标量单元、矢量单元以及专用的矩阵乘法加速指令（Matrix Core），用于支持AI混合精度计算。与NVIDIAGPU显著不同的是，DCU的基本调度单位是波前（Wavefront），包含64个线程，而NVIDIA的Warp为32线程。这一差异对算子设计产生了重要影响：若算法仍沿用32线程的粒度，直接移植到DCU上可能导致线程空闲或同步错误，从而降低硬件利用率。

在存储架构方面，海光DCU采用多级存储层次结构，其中片上共享内存被称为LDS（LocalDataShare）。LDS通常被划分为32个Bank，每个Bank宽度为4字节。由于DCU的Wavefront宽度为64，而LDS的Bank数量仅为32，在一次并发访问中，发生Bank冲突的概率和严重程度显著高于NVIDIA平台。例如，在步长为32的跨步访问模式下，NVIDIA平台可能无冲突，而DCU则会产生严重的串行化延迟。这一硬件特性正是本文提出AccessScatter优化策略的主要动因。

在编程模型方面，为了支撑DCU的应用生态，海光推出了全栈式异构计算开发包DTK（DCU Toolkit）。该工具包对标NVIDIA CUDA Toolkit，基于AMD ROCm开源生态进行深度定制与优化，旨在构建连接上层应用与底层硬件的统一桥梁。在编程接口层面，DTK以HIP为核心编程语言，其语法高度兼容CUDA并支持hipify代码转换，底层执行语义则严格适配DCU的Wavefront64架构。在计算库支持上，DTK集成了rocBLAS、MIOpen等经过手工汇编优化的官方加速库，但受限于算子融合能力不足与版本迭代滞后等问题，这也促使本文力图通过编译优化超越该基准。在指令集架构方面，DUC在AMD的CDNA指令集的基础上进行了指令的拓展，且部分设备自定义硬件标识符，无法通过LLVM生态直接找到对应的指令集架构进行编译，这进一步增加了底层编译与优化的复杂性。

\section{Transformer模型架构与计算特性}

\subsection{Transformer架构综述}
Transformer 模型最早由 Vaswani 等人在 2017 年的经典论文《Attention Is All You Need》中提出。与传统的循环神经网络（RNN）和卷积神经网络（CNN）不同，Transformer 彻底抛弃了循环与卷积结构，完全基于注意力机制来捕捉序列中长距离的依赖关系。这种设计不仅解决了长序列训练中的梯度消失问题，更极大地提升了模型的并行计算能力，使其能够充分利用现代 GPU 等异构硬件的计算资源。
\begin{figure}[htbp]
\centering
\includegraphics[width=0.5\textwidth]{2_5}
\caption{Transformer架构}\label{2_5}
\vspace{\baselineskip}
\end{figure}
如图~\ref{2_5}~所示为标准的 Transformer 架构遵循经典的编码器-解码器（Encoder-Decoder）范式，其均由若干个结构相同的子层级联而成，通常记为 N 层堆叠结构。每一层内部通过残差连接（Residual Connection）和层归一化（Layer Normalization）进行稳定训练，从而在深层网络中有效缓解梯度消失和数值不稳定问题。

Transformer需显式引入位置信息，通过将输入符号嵌入向量与位置编码（固定正弦-余弦形式或可学习参数）逐元素相加，得到最终输入表示。编码器每层包含多头注意力机制与前馈全连接网络（Feed-Forward Network, FFN）：MHA通过并行多个注意力头，将输入映射为查询、键、值向量并计算相关性，拼接输出后线性变换；FFN由两层线性变换与非线性激活函数组成，是模型主要计算密集型模块之一。解码器与编码器结构相似，额外加入编码器—解码器注意力模块，且通过掩码多头自注意力避免自回归生成时的信息泄漏。计算上，模型以矩阵乘法和注意力机制为主，注意力机制在长序列下具有$O(L^{2})$复杂度，对硬件存储要求高，各模块的数据依赖的特性为优化提供空间，也对异构硬件加速和编译器优化提出了更高要求。

\subsection{模型关键算子介绍}
在 Transformer 模型中，绝大部分计算负载集中于矩阵乘法与注意力机制两个核心算子。二者既构成模型的基础运算单元，又在计算量与访存开销上占据主导地位，是影响模型推理性能的关键因素，如图\ref{2_6}分别展示了矩阵乘法与注意力机制的计算流程示意。因此，针对这两个算子的高性能实现具有重要研究价值。
\begin{figure}[htbp]
\centering
\includegraphics[width=0.9\textwidth]{2_6}
\caption{矩阵乘法（左）和注意力机制（右）计算流程示意图}\label{2_6}
\vspace{\baselineskip}
\end{figure}


首先，矩阵乘法是 Transformer 中最基础且最频繁调用的线性代数操作。其计算形式可表示为：
\begin{equation}\label{matmul}
C_{ij} =\sum_{k=0}^{K-1} A_{ik}B_{kj}
\end{equation}
其中，$A\in R^{M\times K}$、$B\in R^{K\times N}$，输出矩阵$C\in R^{M\times N}$ 。该计算本质为三重嵌套循环结构，是典型的计算密集型算子。在 Transformer 模型中，Matmul 广泛存在于注意力机制内部，用于计算$QK^{T}$与 Attention 输出与$V$的乘积，以及前馈神经网络（FFN）中的线性变换层和输入投影与输出投影等全连接操作中，由于大模型通常具有较高的隐藏维度和批大小，Matmul 在整个推理阶段占据主要 FLOPs。与此同时，其性能高度依赖于分块策略、缓存复用及并行映射方式，因此成为深度学习编译优化的核心对象。

其次，注意力机制是 Transformer 区别于传统序列模型的核心创新，其功能是为序列中每个位置动态建模与其他位置之间的依赖关系。最常用的形式为缩放点积注意力（Scaled Dot-Product Attention），其计算流程可表示为：
\begin{equation}\label{attention}
Attention \left ( Q,K,V \right ) = Softmax \left ( \frac{QK^{T} }{\sqrt{d_{k} } }  \right ) 
\end{equation}
其中$Q$、$K$和$V$分别为查询、键和向量值，$d_{k}$为键向量维度。从计算流程上看，Attention 可分为三个阶段，首先计算相关性矩阵$S=QK^{T}$（矩阵乘法操作），然后
对每一行执行缩放与 Softmax 归一化，最后将归一化结果与$V$相乘得到输出结果。在多头注意力结构中，上述过程在多个子空间中并行执行，随后拼接并线性映射。该机制使模型能够从不同表示子空间捕获多样化的语义关系。
Attention 在模型结构中的位置主要位于编码器与解码器层内部，是建模全局依赖关系的关键模块。在长序列场景下，其时间与空间复杂度均为$O(L^{2})$，对显存带宽与片上缓存提出了较高要求。特别是中间生成的相关性矩阵规模较大，若频繁读写全局内存，将严重制约整体性能。

最后，Matmul 与 Attention 计算规模大、算术强度高，数据复用潜力显著但对访存模式敏感，且性能受并行粒度与存储层次设计影响明显。Attention 由多组 Matmul 与逐元素运算构成，是结构复杂、数据依赖明确的复合算子。因此，在编译层面对二者统一建模与协同优化，是提升 Transformer 推理性能的关键。

\section{机器学习编译与MLIR基础}

\subsection{机器学习编译器概述}
随着深度学习模型复杂度的不断提升与硬件后端的日益多样化，传统的基于库的算子实现方式逐渐显露出局限性。为了解决模型与硬件之间高效映射的难题，机器学习编译器应运而生。机器学习编译器是一种专门用于将高层次的深度学习模型描述转换为特定硬件上可执行的高性能机器代码的系统软件。其核心目标是弥合上层算法框架与底层异构硬件之间的语义鸿沟，在减轻人工优化负担的同时，实现代码的自动化生成与性能调优。如图\ref{2_7}所示，展示了传统编译器与机器学习编译器在三个编译阶段上的差异，因此一个成熟的机器学习编译器架构包含以下三个主要阶段：

\begin{figure}[htbp]
\centering
\includegraphics[width=0.85\textwidth]{2_7}
\caption{传统编译器（右）和机器学习编译器（左）的编译流程对比}\label{2_7}
\vspace{\baselineskip}
\end{figure}

\begin{enumerate}
\item 前端（Frontend）与图级优化：编译器首先将深度学习框架定义的计算图转换为高层中间表示。在此阶段，编译器主要执行与硬件无关的优化，例如算子融合、代数化简和死代码消除。算子融合通过将多个计算密集型算子与访存密集型算子合并执行，能够显著减少全局显存的访问开销。
\item 算子级优化与调度：这是编译器的核心环节。在此阶段，编译器将计算图进一步降级为张量级或循环级中间表示。以 TVM 、Triton 等代表性系统为例，它们通过构建搜索空间或使用模板化生成机制，自动搜索最优的循环切分、向量化和并行调度策略。这一过程旨在替代传统的手工编写 CUDA 内核，以适应不同硬件的存储层级与计算特性。
\item 后端（Backend）与代码生成：经过优化的中间表示最终被传递给后端，生成面向特定目标硬件的源代码（如 CUDA、LLVM IR 或 C++），并编译为二进制可执行文件。
\end{enumerate}

尽管现有的深度学习编译器在一定程度上实现了多后端支持与自动化优化，但正如本研究前文所述，许多现有如 TVM等诸多系统在中间表示层面缺乏对 GPU 线程块级结构的直接刻画，难以灵活表达复杂的协同优化模式，这为 Transformer 类大模型的高性能算子生成带来了挑战。

\subsection{MLIR核心设计原理}
MLIR 是由Google提出并主导构建的新型编译器架构，旨在应对深度学习与异构计算领域日益严重的编译器基础设施碎片化挑战。与采用单一抽象层级中间表示的传统编译框架不同，MLIR 提出了以可扩展、多层次中间表示为核心的统一编译框架。如图\ref{2_8}所示，展示了使用 Affine 和 Std 方言的多项式乘法的MLIR通用表示，其进一步表现了MLIR设计思想强调在统一的编译流程中，允许不同抽象层级、不同领域语义的中间表示共存与协同，从而支持从高层算法描述到底层硬件指令的渐进式语义转换与优化，在提升优化能力的同时，降低了跨平台编译系统的工程复杂度。
\begin{figure}[htbp]
\centering
\includegraphics[width=0.72\textwidth]{2_8}
\caption{MLIR的通用表示}\label{2_8}
\vspace{\baselineskip}
\end{figure}

首先，MLIR的方言机制允许开发者根据特定领域需求自定义操作、类型和属性，构建了一套高度灵活的中间表示体系。这种设计在横向维度上实现了不同方言的共存与协同，例如处理线性代数的 Linalg 方言可与 GPU 并行计算方言在同一模块中协同工作，形成领域间的解耦；在纵向维度上，MLIR 提供了一系列内置方言，覆盖了从高层图表示TOSA、中层循环结构Affine、SCF到底层硬件抽象LLVM IR的完整抽象层级，为渐进式编译优化提供了结构化支撑。这种多层级、多领域的方言体系，使得编译器能够精准刻画不同阶段的计算语义，为复杂优化提供了丰富的上下文信息。

其次，MLIR 通过逐级降级与转换机制的 Lowering 流程将高层抽象逐步细化为低层表示，这种“小步快跑”的渐进式转换模式相比传统编译器的一步到位式降级具有显著优势。在转换过程中，如循环边界、张量形状、并行语义等高层语义信息得以保留，使编译器能够在不同抽象层级执行针对性的优化，例如在循环层级进行循环融合、在张量层级进行算子融合，而不会因过早降级导致语义丢失。此外，这种机制还促进了优化逻辑的复用，不同深度学习框架可以共享中间层次的方言定义和优化 Pass，避免了重复开发，提升了整个生态的工程效率。

最后，MLIR 采用基于操作定义规范（ODS）和声明式重写规则（DRR）的定义方法，通过 TableGen 技术来定义方言操作和重写规则，这极大地提升了编译器扩展的开发效率与可维护性。ODS允许开发者以声明式方式定义操作的操作数、结果、属性等元信息，并自动生成类型检查、序列化等基础设施代码。DRR则通过模式匹配和重写规则描述优化转换逻辑，使优化 Pass 的编写更加直观且易于验证。这种声明式方法不仅确保了编译器在扩展时的类型安全与结构一致性，还降低了新方言和优化规则的学习成本，为 MLIR 生态的快速演进提供了技术保障。

正是基于上述设计原则，MLIR 成为构建面向 Transformer 等复杂模型的高性能算子生成框架的重要基础。通过在 MLIR 之上引入面向块级并行结构的自定义 Dialect，可以在保持高层算子语义清晰的同时，精确描述线程块级计算与访存行为，为后续跨平台优化与代码生成提供坚实支撑。

\subsection{基于MLIR的代码生成流程}
基于 MLIR 的代码生成本质上是一个依托多抽象层级中间表示，通过一系列结构化的转换与优化 Pass，将高层计算模型逐步降级并最终映射至目标硬件的过程。与传统编译器采用单一中间表示进行“直接降级”的路径不同，MLIR 的分层设计允许在编译的不同阶段保留最合适的语义信息，从而实现优化灵活性与可扩展性的显著提升。如图\ref{2_9}所示，整个流程可系统性地划分为高层语义建模、中层结构化表示、低层硬件映射与后端代码生成四个核心阶段，其中将模型优化集成于多层IR的Lowering中，且各级IR之间可混合使用MLIR的方言，极大的提高了模型优化的灵活度。

在高层语义建模阶段，输入通常源自深度学习框架的计算图前端或者是模型脚本。该阶段的核心目标是将算子及其数据依赖关系转换为 MLIR 中表达“计算语义是什么”的高层 Dialect 表示，例如张量级操作与算子级数据流。此类表示便于在抽象维度上进行算子融合、代数变换等全局优化，同时避免过早引入硬件相关细节，为后续优化保留了完整的语义搜索空间。
随着编译进程推进，流程进入中层结构化表示与调度表达阶段。高层算子语义在此被展开为更具结构性的循环、分块与并行映射表示，以回答“计算如何被组织与调度”的问题。此阶段通常借助 Dialect 转换与定制化 Pass，将张量计算映射为多级循环结构，并建立计算任务与线程层次间的抽象关系。中层 IR 在保留结构化语义的同时，引入了对存储层级、并行粒度与访存模式的初步刻画，为后续硬件映射奠定基础。

在面向硬件的 Lowering 与资源映射阶段，编译器的任务是将中层表示中的抽象并行结构与存储模型，映射到具体硬件的执行模型与资源体系上。这一过程通过逐步引入更低层级的 Dialect，显式处理线程映射、地址空间、同步语义及寄存器使用等硬件细节。MLIR 的渐进式 Lowering 机制允许多个阶段性 Pass 协同工作，使得通用流程可被不同硬件平台复用，仅在关键节点嵌入平台相关的定制化转换。

最终，在后端代码生成阶段，经过充分降级与硬件映射的 IR 被转换为 LLVM 等传统后端所支持的低层表示，进而生成可执行的目标代码。由于大部分结构化优化已在 MLIR 层面完成，此后端主要承担指令选择、寄存器分配等经典编译任务，实现了高层算子优化与底层指令生成的责任分离，提升了编译流程的稳定性和可维护性。

整体而言，基于 MLIR 的代码生成流程体现了多级 IR 渐进式映射的核心优势：在不同编译阶段保留最有价值的结构化信息、以模块化 Pass 实现高度可扩展的优化、并在复用通用流程的同时灵活适配异构硬件平台。这一特性使 MLIR 成为支撑面向 Transformer 等复杂模型的高性能算子生成框架的理想基础设施，通过引入面向线程块等细粒度硬件抽象的自定义中间表示，可进一步强化对计算调度与数据搬运的编译表达能力

\section{本章小结}
本章首先介绍了异构计算加速平台的发展现状，重点分析了主流的 NVIDIA GPU 与国产 海光 DCU 的硬件架构及其编程模型。通过对比发现，尽管两者均遵循 SIMT 执行模型，但在基本调度单位（Warp vs. Wavefront）、存储层级特性（LDS Bank 冲突）等方面存在显著差异，这些差异构成了跨平台算子优化的底层挑战。

其次，本章深入剖析了 Transformer 模型 的架构组成与计算特性。通过对矩阵乘法与注意力机制两个核心算子的详细分析，指出了大模型推理过程中存在的计算密集与访存密集交织的特点，明确了针对这些关键算子进行深度优化的必要性。

最后，本章探讨了 机器学习编译器 的基本原理，并重点介绍了 MLIR 的核心设计思想。MLIR 通过多层级方言机制与渐进式降级流程，为解决异构硬件生态碎片化、提升算子生成效率提供了先进的基础设施。

综上所述，深入理解异构硬件的微架构特性、掌握 Transformer 算子的数据流向，并利用 MLIR 的分层编译技术，是构建高性能、跨平台算子优化框架的理论基石。本章的研究为后续章节提出针对性的编译优化策略与自定义 Dialect 设计奠定了坚实的理论支撑。
