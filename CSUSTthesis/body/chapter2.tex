\section{异构计算加速平台}
\subsection{异构计算概述}
% 随着深度学习技术的爆发，特别是基于 Transformer 的大语言模型参数量达到万亿规模，传统以 CPU 为核心的同构计算体系已无法满足 AI 任务对计算和访存的需求，面向人工智能的异构计算因此成为支撑现代深度学习模型训练与推理的基础设施范式。深度神经网络和 Transformer 模型的高度规则性和数据并行性，促使计算架构从“以控制为中心”的通用 CPU 转向“以数据为中心”的异构协同模式，其中CPU负责逻辑控制、数据预处理与调度，而加速器则专注执行矩阵乘法、卷积及注意力等核心算子\cite{jouppi2017datacenter}。
深度学习技术发展很快，尤其是 Transformer 大语言模型参数量达到万亿规模，传统以 CPU 作为核心的同构计算模式已经难以满足 AI 任务在计算和访存上的需求，面向人工智能的异构计算因此变成支撑现代深度学习模型训练和推理的基础设施方式。深度神经网络和 Transformer 模型表现出高度规则化及数据并行化的特征，计算架构由此从以控制为中心的通用处理器 CPU，转变为以数据为中心的异构协同计算模式。CPU 在该模式中主要负责逻辑控制、数据预处理和任务调度，加速器则专门进行矩阵乘法、卷积、注意力等主要算子的执行\cite{jouppi2017datacenter}。

% 为争夺大模型时代的算力主导权，全球半导体厂商推出了多种技术路线的异构加速平台，形成了多样化但生态割裂的局面。主流平台主要包括通用图形处理器（GPGPU）路线和专用人工智能芯片（ASIC）路线\cite{jouppi2021ten}。以 NVIDIA GPU 为代表的 GPGPU ，凭借其 CUDA 生态成为当前最主流和成熟的硬件载体。AMD GPU 及国产的海光 DCU 也采用类似架构，是构建国产自主算力底座的关键力量。ASIC 路线则包括华为昇腾 NPU、Google TPU 和寒武纪 MLU 等，针对深度学习计算进行定制化设计，以追求极致能效比。
% 然而，不同平台在指令集架构、微架构设计、存储层级和编程模型上存在显著差异，这导致上层深度学习框架难以通过统一后端充分发挥各类硬件的性能。在众多方案中，NVIDIA GPU代表了工业界的性能标杆与优化基准，而海光DCU则体现了国产化替代的主流方向与典型架构。深入理解这两类代表性硬件的架构及编程模型，是分析算子性能瓶颈、设计跨平台编译优化策略的前提。
为争取大模型时代的算力主导权，全球半导体厂商推出多种技术路线的异构加速平台，形成多样但生态割裂的局面。主流平台主要包括通用路线（GPGPU）和专用人工智能芯片（ASIC）路线\cite{jouppi2021ten}。具有 CUDA 生态的 NVIDIA GPU 这种 GPGPU，已经成为当下主流且成熟的硬件载体。AMD GPU 和国产海光 DCU 也使用相似的架构，它们是形成国产自主算力基础的重要力量。ASIC 路线包括华为升腾 NPU、Google TPU 和寒武纪 MLU 等，专门对深度学习计算进行定制，以获得最好的能效比。不过，各类平台在指令集架构、微架构设计、存储层级和编程模型上存在明显差异，上层深度学习框架因此很难通过统一后端充分利用不同硬件的性能。在各种方案中 NVIDIA GPU 代表工业界性能的基准和优化标准，海光 DCU 体现国产替代的主要方向、典型架构。分析算子性能瓶颈、设计跨平台编译优化策略，都需要以理解这两类代表性硬件的架构及编程模型为前提。

在编程模型层面，CUDA 采用单指令多线程模型，通过大规模线程并行来掩盖访存延迟。如图 \ref{2_3} 所示,在 NVIDIA GPU 编程模型中，其线程组织呈现为“网格-线程块-线程”的三级逻辑抽象，其中线程簇（Cluster）层级为 Hopper 架构后特有抽象层级，不属于传统编程模型的层次结构。
% 在同一个 Cluster 的 Block 可进行数据通信，减少对 HBM 的访问。硬件执行的基本单位是线程束（Warp），由32个线程组成。同一 Warp 内的线程以锁步方式执行相同指令，若出现条件分支会导致分支发散，严重损害执行效率。线程块是资源分配与调度的基本单位，被映射到 SM 上执行。SM 内的 Warp 调度器通过动态上下文切换机制，在等待数据的 Warp 和就绪计算的 Warp 之间快速切换，以实现计算与访存的重叠，从而隐藏延迟。因此，维持足够的占用率，即在 SM 上驻留充足的活跃 Warp 以保持计算流水线持续饱和，是发挥 GPU 极致性能的核心原则\cite{baghsorkhi2010adaptive}。
处于同一个 Cluster 内的 Block 可以进行数据通信，这样能够减少对 HBM 的访问。线程束（Warp）由 32 个线程组成，是硬件执行的基本单位。同一个 Warp 内部的线程利用锁步模式运行相同指令，条件分支的出现会引起分支发散，对于执行效率造成严重影响。线程块作为资源分配和调度的主要单位，会被映射到 SM 上以进行执行。SM 中的 Warp 调度器利用动态上下文切换方式，在等待数据的 Warp 和准备计算的 Warp 之间进行快速切换，来达到计算与访存重叠的​​目的，进而掩盖延迟。因而，保持足够的占用率，也就是在 SM 上驻留足够多的活跃 Warp，从而使计算流水线维持饱和状态，是让 GPU 达到最佳性能的重要原则\cite{baghsorkhi2010adaptive}。

% 当算子启动时，全局调度器将 Grid 中的线程块分配给各个 SM。在 SM 内部，Warp 调度器负责从处于就绪状态的 Warp 中选取指令进行发射。这种设计通过上下文切换机制实现了延迟隐藏：当某个 Warp 正在等待全局显存的数据时，调度器会立即切换到另一个已有数据准备好的 Warp 进行计算。因此，为了充分利用 GPU 的算力，必须保持足够高的占用率，即在 SM 上驻留足够多的活跃 Warp，以确保计算流水线永不停歇。总之，深入理解 NVIDIA GPU 从 SM 微架构、多级存储到 SIMT 执行模型的软硬件协同设计原理，是进行底层算子性能分析与优化不可或缺的基础。
算子开始运行之后，全局调度器会把网格（Grid）里的线程块分配给各个 SM。在 SM 内部，Warp 调度器从处于就绪状态的 Warp 中选择指令进行发射。该设计利用上下文切换机制来达到延迟隐藏的目的，当某个 Warp 在等待全局显存数据时，调度器会立刻切换到另一个数据已经准备好的 Warp 进行计算。所以，为了把 GPU 的计算资源用好，必须保持较高的占用率，也就是在 SM 上驻留足够多的活跃 Warp，来让计算流水线一直运行。充分了解 NVIDIA GPU 在 SM 微架构、多级存储、SIMT 执行模型上的软硬件协同设计原理，构成了对底层算子性能进行分析和优化的必要基础。

\subsection{海光 DCU 架构及编程模型}
% 海光 DCU 是国产异构计算领域的代表性硬件平台，广泛应用于高性能计算中心与大规模 AI 智算集群。其设计理念继承了 GPGPU 的大规模并行架构，采用与 NVIDIA 类似的 SIMT 执行模型，但在微架构实现与资源组织方式上存在显著差异，这些差异成为跨平台算子优化的关键考量因素。

% 在硬件架构方面，海光 DCU 获得了 AMD 的 CDNA 架构\cite{amd_cdna}的技术授权，因此其架构特征与 AMD CDNA 架构基本相似，其核心计算模块是计算单元（Compute Unit, CU），功能上对应于 NVIDIA 的 SM。如图~\ref{2_4}~为其 CU 的架构图，其中每个 CU 内部集成了标量单元、矢量单元以及专用的矩阵乘法加速指令（Matrix Core），用于支持 AI 混合精度计算。与 NVIDIA GPU 显著不同的是，DCU 的基本调度单位是波前（Wavefront），包含64个线程，而 NVIDIA 的 Warp 为32线程。这一差异对算子设计产生了重要影响：若算法仍沿用32线程的粒度，直接移植到 DCU 上可能导致线程空闲或同步错误，从而降低硬件利用率。
海光 DCU 在硬件架构上得到了AMD CDNA 架构的技术授权，所以它的架构特点和 AMD CDNA 架构\cite{amd_cdna}大体一样，其中心计算模块是计算单元（Compute Unit, CU），在功能上同 NVIDIA 的 SM 相对应。图 \ref{2_4} 表达了 CU 的架构。CU 内部的标量单元、矢量单元、专用矩阵乘法加速单元（Matrix Core）被集成在一起，以此支持 AI 混合精度计算。DCU 的基本调度单位是包括 64 个线程的波前（Wavefront），这与 NVIDIA GPU 不同，NVIDIA 的 Warp 有32个线程。若接着使用 32 线程粒度，直接把算法移植到 DCU 会引起线程空闲或者同步错误，进而让硬件利用率下降。

% 在存储架构方面，海光 DCU 采用多级存储层次结构，其中片上共享内存被称为 LDS（Local Data Share）。LDS 通常被划分为32个 Bank，每个 Bank 宽度为4字节。由于 DCU 的 Wavefront 宽度为64，而 LDS 的 Bank 数量仅为32，在一次并发访问中，发生 Bank 冲突的概率和严重程度显著高于 NVIDIA 平台。例如，在步长为32的跨步访问模式下，NVIDIA 平台可能无冲突，而 DCU 则会产生严重的串行化延迟。这一硬件特性正是本文提出 AccessScatter 优化策略的主要动因。
海光 DCU 在存储架构上使用了多级存储层次结构，并且把片上共享内存叫做 LDS（Local Data Share）。LDS 通常分为 32 个 Bank，每个 Bank 的宽度为 4 字节。因为 DCU 的 Wavefront 宽度是 64，而 LDS 的 Bank 数量只有 32，在一次并发访问中，发生 Bank 冲突的概率和严重程度比 NVIDIA 平台更高。例如，在步长为 32 的跨步访问模式情况下，NVIDIA 平台不会产生冲突，但是 DCU 会出现串行化延迟。Access Scatter 优化策略的主要原因是硬件特性。

% 在编程模型方面，为了支撑 DCU 的应用生态，海光推出了全栈式异构计算开发包 DTK（DCU Toolkit）。该工具包对标 NVIDIA CUDA Toolkit，基于 AMD ROCm 开源生态进行深度定制与优化，旨在构建连接上层应用与底层硬件的统一桥梁。在编程接口层面，DTK 以 HIP 为核心编程语言，其语法高度兼容 CUDA 并支持 hipify 代码转换，底层执行语义则严格适配 DCU 的 Wavefront64 架构。在计算库支持上，DTK 集成了 rocBLAS、MIOpen 等经过手工汇编优化的官方加速库，但受限于算子融合能力不足与版本迭代滞后等问题，这也促使本文力图通过编译优化超越该基准。在指令集架构方面，DUC 在 AMD 的 CDNA 指令集的基础上进行了指令的拓展，且部分设备自定义硬件标识符，无法通过 LLVM\cite{lattner2004llvm}生态直接找到对应的指令集架构进行编译，这进一步增加了底层编译与优化的复杂性。
在编程模型上，为支持 DCU 的应用生态，海光发布了全栈式异构计算开发包 DTK（DCU Toolkit）。它对标 NVIDIA CUDA Toolkit，而且根据 AMD ROCm 开源生态所进行的定制优化，尝试作为连接上层应用和底层硬件的统一桥梁。就编程接口而言，DTK 把 HIP 当作主要编程语言，它的语法和 CUDA 非常相似，并且可以对代码进行 hipify 转换，而底层执行语义则完全适用于 DCU 的 Wavefront64 架构。在计算库支持上，DTK 集成了 rocBLAS 和 MIOpen 等官方加速库，这些库都经过了手工汇编优化。但是，因为算子融合能力有些不足并且版本迭代存在滞后，本文通过编译优化方式尝试来超越该基准。在指令集架构这方面，DUC 对 AMD 的 CDNA 指令集进行了扩展，部分设备也自行定义了硬件标识符，LLVM\cite{lattner2004llvm} 生态很难直接找到对应的指令集架构来完成编译工作，这直接增加了底层编译和优化的复杂程度。

\section{Transformer 模型架构与计算特性}

\subsection{Transformer 架构综述}
Transformer 模型最早由 Vaswani 等人在 2017 年的经典论文《Attention Is All You Need》\cite{vaswani2017attention}中提出。与传统的循环神经网络（RNN）和卷积神经网络（CNN）不同，Transformer 彻底抛弃了循环与卷积结构，完全基于注意力机制来捕捉序列中长距离的依赖关系。这种设计不仅解决了长序列训练\cite{press2021train}中的梯度消失问题，更极大地提升了模型的并行计算能力，使其能够充分利用现代 GPU 等异构硬件的计算资源。

\end{figure}
% 如图~\ref{2_5}~所示为标准的 Transformer 架构遵循经典的编码器-解码器（Encoder-Decoder）范式\cite{xiong2020layer}，其均由若干个结构相同的子层级联而成，通常记为 N 层堆叠结构。每一层内部通过残差连接\cite{he2016deep}（Residual Connection）和层归一化\cite{ba2016layer}（Layer Normalization）进行稳定训练，从而在深层网络中有效缓解梯度消失和数值不稳定问题。

% Transformer 需显式引入位置信息，通过将输入符号嵌入向量与位置编码逐元素相加，得到最终输入表示\cite{su2024roformer}。编码器每层包含多头注意力机制与前馈全连接网络\cite{bebis2002feed}（Feed-Forward Network, FFN）：MHA 通过并行多个注意力头，将输入映射为查询、键、值向量并计算相关性，拼接输出后线性变换；FFN 由两层线性变换与非线性激活函数组成，是模型主要计算密集型模块之一。解码器与编码器结构相似，额外加入编码器—解码器注意力模块，且通过掩码多头自注意力避免自回归生成时的信息泄漏。计算上，模型以矩阵乘法和注意力机制为主，注意力机制在长序列下具有$O(L^{2})$复杂度，对硬件存储要求高，各模块的数据依赖的特性为优化提供空间，也对异构硬件加速和编译器优化提出了更高要求。

Transformer 需要显式地引入位置信息，通过把输入符号嵌入向量同位置编码进行逐元素相加，来得到最终的输入表示\cite{su2024roformer}。编码器每层包括多头注意力机制和前馈全连接网络\cite{bebis2002feed}（Feed-Forward Network, FFN）。多头注意力机制将输入映射为查询、键、值向量并计算相关性，并行多个注意力头，然后拼接输出并进行线性变换。FFN 包括两层线性变换和非线性激活函数，是模型里计算量非常大的模块。解码器和编码器结构比较类似，并且加入编码器—解码器注意力模块，还通过掩码多头自注意力来防止自回归生成时的信息泄漏。计算上，模型以矩阵乘法和注意力机制为主，注意力机制在长序列下具有 $O(L^{2})$ 复杂度，对硬件存储要求高，各模块的数据依赖的特性为优化提供空间，也对异构硬件加速和编译器优化提出了更高要求。

\subsection{模型关键算子介绍}
% 在 Transformer 模型中，绝大部分计算负载集中于矩阵乘法与注意力机制两个核心算子。二者既构成模型的基础运算单元，又在计算量与访存开销上占据主导地位，是影响模型推理性能的关键因素，如图\ref{2_6}分别展示了矩阵乘法与注意力机制的计算流程示意。因此，针对这两个算子的高性能实现具有重要研究价值\cite{wang2020linformer}。
在 Transformer 模型中，大部分计算负载集中在矩阵乘法和注意力机制这两个主要算子上。这两部分构成了模型的基本运算单元，它们在计算量和访存开销方面也占据主要位置，是影响模型推理效果的重要因素。图 \ref{2_6} 显示了矩阵乘法和注意力机制运算过程的示意图。所以，对这两个算子的高性能实现进行具有重要研究价值。

\centering
\includegraphics[width=0.9\textwidth]{2_6}
\caption{矩阵乘法（左）和注意力机制（右）计算流程示意图}\label{2_6}

首先，矩阵乘法是 Transformer 中最基础且最频繁调用的线性代数操作。其计算形式可表示为：
\begin{equation}\label{matmul}
C_{ij} =\sum_{k=0}^{K-1} A_{ik}B_{kj}
\end{equation}
其中，$A\in R^{M\times K}$、$B\in R^{K\times N}$，输出矩阵$C\in R^{M\times N}$ 。公式表明矩阵乘法是典型的计算密集型算子。在 Transformer 模型中，Matmul 广泛存在于注意力机制内部、前馈神经网络（FFN）中的线性变换层以及全连接操作中。大模型通常具有较高的隐藏维度和批大小，所以 Matmul 在整个推理阶段占据主要开销。Matmul 的性能高度依赖于分块策略、缓存复用及并行映射方式，因此成为深度学习编译优化的核心对象。

其次，注意力机制是 Transformer 模型的核心创新，其功能是为序列中每个位置动态建模与其他位置之间的依赖关系。其中缩放点积注意力（Scaled Dot-Product Attention）计算流程可表示为：
\begin{equation}\label{attention}
Attention \left ( Q,K,V \right ) = Softmax \left ( \frac{QK^{T} }{\sqrt{d_{k} } }  \right ) 
\end{equation}
其中 $Q$、$K$ 和 $V$ 分别为查询、键和向量值，$d_{k}$ 为键向量维度。从计算流程上看，MHA 可分为三个阶段，首先计算相关性矩阵 $S=QK^{T}$，然后
对每一行执行缩放与 Softmax 归一化，最后将归一化结果与 $V$ 相乘得到输出结果。
% 在多头注意力结构中，上述过程在多个子空间中并行执行，随后拼接并线性映射。该机制使模型能够从不同表示子空间捕获多样化的语义关系。
% Attention 在模型结构中的位置主要位于编码器与解码器层内部，是建模全局依赖关系的关键模块。在长序列场景下，其时间与空间复杂度均为$O(L^{2})$，对显存带宽与片上缓存提出了较高要求。特别是中间生成的相关性矩阵规模较大，若频繁读写全局内存，将严重制约整体性能\cite{beltagy2020longformer}。
多头注意力结构里，前面提到的过程在多个子空间里同时进行，然后再把结果拼接起来做线性映射。该模式使模型能够从各个表示子空间中获得多样化的语义关系。MHA 在模型结构里主要处在编码器和解码器层中，该模块对建模全局依赖关系起着主要作用。处理长序列时，时间复杂度和空间复杂度都会达到 $O(L^{2})$，这对显存带宽和片上缓存的要求较高。特别是中间生成的相关性矩阵规模较大，如果频繁读写全局内存，会严重制约整体性能\cite{beltagy2020longformer}。

\section{机器学习编译与 MLIR 基础}

\subsection{机器学习编译器概述}
随着深度学习模型复杂度的不断提升与硬件后端的日益多样化，传统的基于库的算子实现方式逐渐显露出局限性。为了解决模型与硬件之间高效映射的难题，机器学习编译器\cite{zhang2023compiler}应运而生。机器学习编译器是一种专门用于将高层次的深度学习模型描述转换为特定硬件上可执行的高性能机器代码的系统软件。其核心目标是弥合上层算法框架与底层异构硬件之间的语义鸿沟，在减轻人工优化负担的同时，实现代码的自动化生成与性能调优。如图 \ref{2_7} 所示，展示了传统编译器与机器学习编译器在三个编译阶段上的差异，因此一个成熟的机器学习编译器架构包含以下三个主要阶段：

\centering
\includegraphics[width=0.85\textwidth]{2_7}
\caption{传统编译器（右）和机器学习编译器（左）的编译流程对比}\label{2_7}
\vspace{\baselineskip}
\end{figure}

\begin{enumerate}
% \item 前端（Frontend）与图级优化：编译器首先将深度学习框架定义的计算图转换为高层中间表示\cite{roesch2018relay, zhong2025flashtensor}。在此阶段，编译器主要执行与硬件无关的优化，例如算子融合\cite{JSJA202505006}、代数化简和死代码消除。算子融合通过将多个计算密集型算子与访存密集型算子合并执行，能够显著减少全局显存的访问开销。
\item 前端和图级优化，编译器先会把深度学习框架定义的计算图转换成高层中间表示\cite{roesch2018relay, zhong2025flashtensor}。编译器在该阶段主要负责执行和硬件无关的优化操作，具体包括算子融合\cite{JSJA202505006}、代数化简、死代码消除。算子融合把多个计算密集型算子和访存密集型算子合在一起进行执行，能够明显减少对全局显存的访问开销。
% \item 算子级优化与调度：这是编译器的核心环节。在此阶段，编译器将计算图进一步降级为张量级或循环级中间表示\cite{feng2023tensorir}。以 TVM 、Triton 等代表性系统为例，它们通过构建搜索空间或使用模板化生成机制，自动搜索最优的循环切分、向量化和并行调度策略。这一过程旨在替代传统的手工编写 CUDA 内核，以适应不同硬件的存储层级与计算特性。
\item 算子级优化和调度，编译器把计算图进行转换，使其变为张量级或者循环级的 IR\cite{feng2023tensorir}。像 TVM 这类典型系统通过建立搜索空间或者使用模板化生成方法，能够自动对循环切分、向量化、并行调度策略进行搜索。这个过程主要用来替代通常的手工编写 CUDA 内核，以此来适应不同硬件的存储层级和计算特性。
\item 后端与代码生成，优化后的 IR 最终被传递给后端，生成面向特定目标硬件的源代码（如 CUDA、LLVM IR 或 C++），并编译为二进制可执行文件。
\end{enumerate}

% 尽管现有的深度学习编译器在一定程度上实现了多后端支持与自动化优化，但正如本研究前文所述，许多现有如 TVM 等诸多系统在中间表示层面缺乏对 GPU 线程块级结构的直接刻画，难以灵活表达复杂的协同优化模式，这为 Transformer 类大模型的高性能算子生成带来了挑战。
虽然现存的深度学习编译器在某种程度上完成了多后端支持和自动化优化，但是正如本文前文所述，包括 TVM 在内的很多系统在中间表示方面上没有直接刻画 GPU 线程块级结构，不容易灵活表达复杂的协同优化模式，给 Transformer 类大模型的高性能算子生成造成了困难。

% 基于 MLIR 的代码生成本质上是一个依托多抽象层级中间表示，通过一系列结构化的转换与优化 Pass，将高层计算模型逐步降级并最终映射至目标硬件的过程。与传统编译器采用单一中间表示进行“直接降级”的路径不同，MLIR 的分层设计允许在编译的不同阶段保留最合适的语义信息，从而实现优化灵活性与可扩展性的显著提升。如图\ref{2_9}所示，整个流程可系统性地划分为高层语义建模、中层结构化表示、低层硬件映射与后端代码生成四个核心阶段，其中将模型优化集成于多层 IR 的 Lowering 中，且各级IR之间可混合使用 MLIR 的方言，极大的提高了模型优化的灵活度。

进行高层语义建模时，输入一般来自深度学习模式的计算图前端或模型脚本。本文该阶段的主要目标是将算子、数据依赖关系转换到 MLIR 之中表达出计算语义含义的这种高层 Dialect 表示，具体包括了张量级操作和算子级数据流。此类别表示方式有利于在抽象方面进行算子融合和代数变换等全局优化，并防止过早引入与硬件相关的细节，从而给后续优化保全了完整的语义搜索空间。

% 随着编译进程推进，流程进入中层结构化表示与调度表达阶段。高层算子语义在此被展开为更具结构性的循环、分块与并行映射表示，以回答“计算如何被组织与调度”的问题。此阶段通常借助 Dialect 转换与定制化 Pass，将张量计算映射为多级循环结构，并建立计算任务与线程层次间的抽象关系。中层 IR 在保留结构化语义的同时，引入了对存储层级、并行粒度与访存模式的初步刻画，为后续硬件映射奠定基础\cite{wang2025loop}。
流程在此阶段进入中层结构化表示及调度表达阶段。高层算子语义被扩展为更结构化的循环、分块和并行映射形式，用来解决计算组织和调度方式的问题。该阶段通常通过 Dialect 转换和定制化 Pass，把张量计算映射为多级循环结构，并且也建立计算任务和线程层次中的抽象关系。中间表示在保留结构化语义的同时，也对存储层级、并行粒度和访存模式进行初步描述，为后续的硬件映射打下基础\cite{wang2025loop}。

% 在面向硬件的 Lowering 与资源映射阶段，编译器的任务是将中层表示中的抽象并行结构与存储模型，映射到具体硬件的执行模型与资源体系上。这一过程通过逐步引入更低层级的 Dialect，显式处理线程映射、地址空间、同步语义及寄存器使用等硬件细节。MLIR 的渐进式 Lowering 机制允许多个阶段性 Pass 协同工作，使得通用流程可被不同硬件平台复用，仅在关键节点嵌入平台相关的定制化转换。
编译器在面向硬件的 Lowering 和资源映射阶段，负责把中层表示里的抽象并行结构、存储模型，映射到具体硬件的执行模型和资源系统上。该过程利用逐步引入层级较低的 Dialect 对硬件细节进行显式处理，包括线程映射、地址空间、同步语义和寄存器使用。MLIR 的渐进式 Lowering 方式让多个阶段性的 Pass 共同进行作用，该通用流程可以在不同硬件平台上被再次使用，仅仅在主要位置放入含有平台特点的个性化转换。

最终，在后端代码生成阶段，经过充分降级和硬件映射的中间表示被转换为 LLVM 等传统后端所支持的低层表示，并生成可执行的目标代码。MLIR 方面已经完成了大部分结构化优化工作，后端主要承担指令选择和寄存器分配等经典编译任务，以此实现高层算子优化与底层指令生成的责任分离，编译流程的稳定性和可维护性由此得到提高。

% 整体而言，基于 MLIR 的代码生成流程体现了多级 IR 渐进式映射的核心优势：在不同编译阶段保留最有价值的结构化信息、以模块化 Pass 实现高度可扩展的优化、并在复用通用流程的同时灵活适配异构硬件平台。这一特性使 MLIR 成为支撑面向 Transformer 等复杂模型的高性能算子生成框架的理想基础设施，通过引入面向线程块等细粒度硬件抽象的自定义中间表示，可进一步强化对计算调度与数据搬运的编译表达能力
