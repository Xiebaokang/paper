\section{矩阵乘法算子代码生成测试}
矩阵乘法作为一种线性代数的基本运算在数学、物理等领域具有十分重要的意义，其不仅可以解决数学领域对于线性方程的求解，而且在人工智能领域作为线性层、注意力机制等计算也同样具有重要意义，因此，本节针对 DeepGen 生成的矩阵乘算子进行多方位的测试是十分具有研究价值的。

对于矩阵乘法的测试，本文首先将 DeepGen 生成的矩阵乘法算子与 NVIDIA 平台和海光平台上的 BLAS 库中矩阵乘法算子进行了对比，选取 M、N、K 维度 4 种不同比例进行对比测试，随后又对比了不同平台上，同样基于 MLIR 开发的代码生成工作--Triton 和经典代码生成项目 TVM 在 M、N、K 维度相等的方阵上进行了对比，且实验均采用每秒浮点计算次数 TFLOPS 作为性能指标，其中与 BLAS 库的对比实验还计算了加速比作为衡量指标。
% 矩阵乘法作为线性代数中的基础运算，在科学计算和深度学习中均具有重要作用。在 Transformer 模型中，线性层、前馈网络以及注意力机制中的核心计算都可归结为矩阵乘法，因此矩阵乘法算子的生成质量会直接影响整体推理性能。基于此，本文针对 DeepGen 生成的矩阵乘法算子进行了系统测试，以验证其在异构平台上的代码生成能力与性能可移植性。

% 按照本文实验设计，首先将 DeepGen 自动生成的矩阵乘法算子分别与 NVIDIA 平台上的 cuBLAS 以及海光平台上的 rocBLAS 进行对比。为了尽可能覆盖不同类型的矩阵形状，实验选取了四组具有代表性的矩阵规模比例，即 $M:N:K=1:1:1$、$1:2:4$、$1:4:2$、$1:1:4$，并在多个尺寸下测试其计算吞吐率。随后，为进一步评估 DeepGen 与现有代码生成工作的差异，本文又在方阵 $M=N=K$ 的配置下，将 DeepGen 与 Triton、TVM 进行了对比。所有实验均以 TFLOPS 作为性能指标，在与 BLAS 库对比时同时计算 DeepGen 相对基线库的加速比，以更直观地反映其性能收益。

如图 \ref{a100_matmul} 所示，在 NVIDIA A100 平台上，DeepGen 在四组不同矩阵形状下整体均能达到与 cuBLAS 接近的性能水平，并在部分规模上实现超越。对于较小规模矩阵，由于计算量有限，线程块调度开销、访存启动开销以及边界处理代价占比较高，因此不同实现之间存在一定波动。随着矩阵规模增大，DeepGen 的性能逐渐稳定，并在多数测试点上与 cuBLAS 持平。尤其在部分大规模配置下，DeepGen 相比 cuBLAS 的性能提升超过 10\%，说明其自动生成代码在计算调度和数据访问组织方面已经具备较高质量。

分析可知，当 $M>10240$ 时，DeepGen 在 A100 上出现较明显的反超现象，这与 cuBLAS 在该规模区间内存在一定性能回落有关。其原因可能在于，cuBLAS 作为面向广泛场景的通用库，内部优化策略更强调整体适用性；而 DeepGen 通过块级分层 IR、Lowering 信息推断和自动调优机制，能够生成更贴合当前矩阵形状与硬件结构的执行方案，因此在特定大规模场景下可以获得更优结果。

其原因在于，DeepGen 并不依赖特定厂商的专有模板，而是通过统一的块级 IR 显式描述矩阵乘法中的分块计算、线程映射和数据搬运关系，再借助跨平台优化 Pass 对共享存储访问、线程块计算顺序和访存离散化等细节进行调优。这类优化有助于减少 Bank 冲突并提升缓存命中率，因此能够在海光平台上取得优于官方库的结果。

在完成与官方 BLAS 库的比较后，本文进一步将 DeepGen 与 Triton、TVM 进行对比，结果如图 \ref{a100_ttd}和图\ref{z100_ttd} 所示。实验采用方阵 $M=N=K$ 配置，以减少形状差异带来的干扰。需要说明的是，部分规模下 TVM 未能搜索到可执行的矩阵乘法算子，因此图中存在空缺点。

从结果来看，DeepGen 在两个平台上的整体性能与 Triton、TVM 处于同一水平，并在部分规模下略有优势。这说明 DeepGen 在保持统一 IR 表达和跨平台生成能力的同时，仍能生成具有较强竞争力的高性能实现。总体而言，无论是与 cuBLAS、rocBLAS 的对比，还是与 Triton、TVM 的比较，DeepGen 在矩阵乘法代码生成上都取得了较理想的效果。该结果验证了前文所设计的分层 IR、优化 Pass 以及自动调优机制的有效性，也为后续注意力机制和端到端推理实验奠定了基础。

\section{注意力机制代码生成测试}
众所周知，Transformer 架构在人工智能的各个领域已经有着不可替代的地位，其核心计算多头注意力机制的优化更是前沿热点。本节中，我们将针对多头注意力机制进行测试对比，我们选取了同样基于 MLIR 生态开发的端到端推理框架--IREE、以 TVM 为后端的开发 GPU Kernel 的领域特定语言工作--Tilelang以及 Triton 和 PyTorch 进行对比测试，其中由于国产海光平台未对 Tilelang 生态进行支持，所以在海光 Z100 平台上只针对除 Tilelang 的其他工作进行对比测试。

在实验中，由于注意力批大小（Batch Size, BS）以及注意力头数（Head Number, HN）均由线程块的 Z 和 Y 维度进行索引，因此文中提及的批大小（BS）均为注意力批大小和注意力头数的乘积。为分析不同输入特征对性能的影响，本节分别开展了序列长度敏感性实验和批大小敏感性实验：前者固定批大小与头维度，考察序列长度变化对性能的影响；后者固定序列长度与头维度，分析批大小变化带来的吞吐率变化。

% Transformer 模型之所以能够在自然语言处理和大模型任务中取得优异效果，多头注意力机制起到了核心作用。与矩阵乘法相比，注意力机制不仅包含 Query、Key、Value 间的矩阵计算，还涉及缩放、Softmax、重排等多个阶段，因此其访存模式更加复杂，也更能反映代码生成框架在算子融合、调度优化和跨平台适配方面的综合能力。基于此，本文对 DeepGen 生成的 MHA 算子进行了系统测试，以验证其在不同平台上的性能表现。

% 按照本文实验设计，选取了 IREE、Tilelang、Triton 以及 PyTorch 作为对比对象。其中，IREE 同样基于 MLIR 生态构建，Tilelang 则面向 GPU Kernel 开发；由于海光 Z100 平台尚未支持 Tilelang，因此在 Z100 上仅与 IREE、Triton 和 PyTorch 进行对比。实验中，由于注意力批大小和注意力头数分别映射到线程块的 Z 维和 Y 维，因此文中提及的批大小 BS 实际表示注意力批大小与头数的乘积。为分析不同输入特征对性能的影响，本节分别开展了序列长度敏感性实验和批大小敏感性实验：前者固定批大小与头维度，考察序列长度变化对性能的影响；后者固定序列长度与头维度，分析批大小变化带来的吞吐率变化。

如图 \ref{a100_attn_fix_bs} 和图 \ref{z100_attn_fix_bs} 所示，本文首先测试了 A100 与 Z100 平台上 MHA 对序列长度变化的敏感性。整体来看，DeepGen 在两个平台上的多数配置下都取得了最高或接近最高的 TFLOPS，尤其在 HD=64 的小头维度场景中优势更为明显。在 A100 平台上，PyTorch 与 Tilelang 的整体性能较为接近，普遍优于 IREE 与 Triton；IREE 的性能随着 HD 增大而有所提升，在较大头维度配置下逐渐超过 Triton，而 Triton 的性能曲线则相对平稳，没有随着参数规模增大而出现明显提升。

从加速比看，DeepGen 在 A100 上相对 PyTorch 的平均加速在 HD=64 时约为 1.46 倍，在 HD=128 时约为 1.11 倍。这种差异与不同头维度下矩阵乘和 Softmax 的分块策略有关。对于 HD=128，现有库实现更容易获得较高效率；而在 HD=64 时，资源更紧张，DeepGen 的融合实现能够更有效地减少中间结果写回并提升计算强度，因此优势更加明显。在 Z100 平台上，随着 BS 增大，DeepGen 相对 PyTorch 的加速比保持在 1.21 倍到 1.24 倍之间，整体较为稳定，说明其优化策略在国产异构平台上同样具有较好的可迁移性。

在完成序列长度敏感性分析后，本文进一步开展了批大小敏感性实验，结果如图 \ref{a100_attn_fix_sl} 和图 \ref{z100_attn_fix_sl} 所示。总体来看，DeepGen 在 A100 和 Z100 平台上的大部分配置下仍然取得了最高 TFLOPS，且在长序列场景下优势更加突出，尤其是在 $SL=16384,HD=64$ 的配置中表现最为明显。这说明随着批大小增加，注意力算子暴露出更高的块级并行度，而 DeepGen 设计的线程块映射与调度方式能够较充分地利用这部分并行性。

对于 A100 平台，IREE 在中大批量下表现出较好的扩展性和稳定性，但整体仍难以在多数配置下超过 DeepGen；Triton 的性能则相对平稳。DeepGen 相对 Triton 的平均加速约为 2.05 倍至 2.85 倍，相对 PyTorch 的平均加速约为 1.06 倍至 1.54 倍。对于 Z100 平台，当批大小较大时采用 $SL=512$ 配置，批大小较小时采用 $SL=16384$ 配置，其加速比约为 1.29 倍至 1.35 倍；而在长序列且 $BS=1$ 时，DeepGen 与 PyTorch 的性能基本相当，随着批大小增加，其优势逐渐显现。

综合图 \ref{a100_attn_fix_bs} 至图 \ref{z100_attn_fix_bs} 可以看出，DeepGen 在不同序列长度、不同头维度和不同批大小条件下都表现出较好的性能鲁棒性。其原因在于，DeepGen 通过分层 IR 对 MHA 的块级计算逻辑进行显式建模，并结合 Lowering 信息推断与优化 Pass，对线程束调度、共享存储访问和寄存器使用等细节进行针对性优化。相比之下，Triton 和 Tilelang 更依赖高级抽象的自动映射方式，在 CUDA Core 这类对细粒度调度高度敏感的场景中，其生成代码的性能容易受到限制；IREE 虽同样基于 MLIR，但采用更通用的编译路径，因此在小头维度场景下优势不够明显。

总体而言，本节实验从序列长度和批大小两个维度验证了 DeepGen 对注意力机制的代码生成能力。实验结果表明，DeepGen 在 NVIDIA A100 与海光 Z100 两个平台上均能在多数配置下取得最优或接近最优的性能，且在长序列、小头维度以及高并行度场景中优势更加突出。这说明本文提出的基于 MLIR 的注意力机制自动代码生成与优化方法是有效的，也进一步验证了前文分层 IR、优化 Pass 和自动调优策略在复杂 Transformer 核心算子中的适用性。

\section{计算精确度验证}\label{precision}
对于矩阵乘法和注意力机制这类 Transformer 模型中的关键算子而言，若编译优化过程中引入了线程重排、访存重组、融合计算或分块规约等变换，虽然可能显著提高执行效率，但也可能因浮点计算顺序变化、并行规约误差累积或中间结果处理方式不同而对最终输出产生影响。因此，在完成性能测试后，本文进一步对 DeepGen 生成算子的数值精度进行了验证，以评估其在异构平台上生成代码的正确性与稳定性。

本节实验分别针对矩阵乘法算子和注意力机制进行测试。对于矩阵乘法，选取了三组方阵规模 $M=N=K$，分别为 512、1024 和 2048；对于注意力机制，则采用 BHSD 形式描述输入规模，其中 B、H、S、D 分别表示批大小、注意力头数、序列长度和头维度，测试配置包括 $(1,12,1024,64)$、$(1,16,2048,64)$ 和 $(1,32,4096,128)$。实验中均以 PyTorch 官方实现的计算结果作为参考基准，并采用相对 L2 范数误差衡量 DeepGen 输出与基准结果之间的差异。

表 \ref{tab:table5} 给出了注意力机制的精度验证结果。可以看到，在 NVIDIA 平台上，三组配置下的误差分别为 $3.869058×10^{-7}$、$6.390584×10^{-7}$ 和 $8.330952×10^{-7}$，同样稳定在 $10^{-7}$ 量级；在海光平台上，误差分别为 $5.230882×10^{-7}$、$2.065086×10^{-6}$ 和 $3.410693×10^{-6}$，也始终处于较低水平。相较于矩阵乘法，注意力机制包含两次矩阵乘、缩放、Softmax 归一化以及输出重组等多个阶段，计算链路更长、数据流更复杂，因此其精度验证更能体现整个编译框架的数值可靠性。实验结果表明，DeepGen 在复杂融合算子场景下同样保持了良好的数值稳定性。

从整体趋势来看，NVIDIA 平台上的误差普遍低于海光平台，且基本稳定在 $10^{-7}$ 量级；海光平台随着问题规模增大，误差略有上升，但最大值仍远低于 $10^{−3}$ 的工业应用常用精度阈值。海光 Z100 平台的误差虽然会随规模增长略有增加，但全尺寸下仍维持在较低范围内，说明该框架在不同硬件平台上均具备良好的数值一致性。造成平台间误差差异的主要原因，通常与底层指令实现、并行规约顺序以及不同硬件后端的浮点执行细节有关，但这类微小差异属于可接受的数值波动范围，并不会影响模型推理结果的有效性。

总体而言，表 \ref{tab:table4} 和表 \ref{tab:table5} 的结果表明，DeepGen 生成的矩阵乘法算子与注意力机制算子在 NVIDIA 和海光两类异构平台上均具有较高的数值精度。无论是基础的矩阵乘法，还是包含多阶段融合的注意力机制，其输出结果都与 PyTorch 保持了高度一致性。这说明本文提出的分层 IR、Lowering 信息推断以及跨平台优化 Pass 在提升性能的同时，并未破坏算子的数学语义和数值正确性，从而验证了 DeepGen 作为自动代码生成框架在工程应用中的可靠性。
